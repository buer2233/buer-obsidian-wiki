### 问题1：多线程、多进程、协程有什么区别与联系？

**考察点：** Python 并发基础、GIL 理解、I/O 密集与 CPU 密集任务选型、自动化测试任务调度能力。

**回答引导：**

1. 先说三者都是并发处理方式，目标都是提升任务处理效率，但调度单位和适用场景不同。
2. 多线程：同一进程内多个线程，共享内存，适合 I/O 密集型任务，如接口请求、文件读写、网络等待。
3. 多进程：多个独立进程，内存隔离，能绕开 CPython GIL，更适合 CPU 密集型任务，但启动和通信成本更高。
4. 协程：单线程内的用户态协作调度，依赖 `async/await`，适合大量网络 I/O、连接等待、日志采集等场景，资源开销低。
5. 结合 Python 说明 GIL：线程不适合提升纯 CPU 计算能力，但对 I/O 密集场景仍然有效。
6. 最后给选型：CPU 密集用多进程，I/O 密集可用多线程或协程，超大并发网络任务优先协程加限流。

**结合个人项目的回答素材：**

- 你简历中有 Python 8 年、pytest、requests、Playwright、Locust、接口自动化和性能测试经验。
- Locust 本身就是基于协程/轻量级并发思想模拟大量用户，这是你性能测试回答可以关联的点。
- AI+接口自动化 SKILL 中，执行 pytest、收集失败日志、抓包分析都属于 I/O 密集型工作，更适合用并发、队列和限流，而不是盲目开大量进程。

**建议回答结构：**

```text
这三者都是并发模型，但层级不一样。
多线程是在一个进程内创建多个线程，内存共享，切换成本比进程低，适合接口请求、文件读写这类 I/O 密集任务。但在 CPython 里有 GIL，纯 CPU 计算不能靠多线程明显提速。
多进程是多个独立进程，内存隔离，稳定性更好，也能绕开 GIL，适合 CPU 密集任务，但进程创建和进程间通信成本更高。
协程是用户态调度，一个线程里通过 async/await 主动让出执行权，适合大量网络 I/O，比如批量请求接口、远程采集日志、异步写文件。
我实际选型时会看任务类型：CPU 密集用多进程，普通 I/O 并发可用线程池，大量连接和等待型任务更倾向 asyncio 协程，并且一定要加并发限制和失败重试。
```

**避坑提醒：**

- 不要说“多线程一定更快”，Python 里要说明 GIL。
- 不要把协程说成真正的多核并行，它是协作式并发。
- 面试官如果追问自动化执行稳定性，要讲限流、超时、重试、队列和资源隔离。

### 问题2：10000 台机器都有一个 log 文件，要收集到本地电脑，本地资源有限，用 Python 怎么处理？

**考察点：** 大规模 I/O 设计、资源受限场景处理、并发限流、日志采集架构、可靠性设计。

**回答引导：**

1. 先说明不能在本地一次性对 10000 台机器全量并发拉取，也不能把所有日志一次性读入内存。
2. 正确思路是分批、限流、流式处理、断点续传、压缩、过滤和落盘。
3. 如果架构允许，优先推荐远端 agent 或现有日志系统：Filebeat/Fluent Bit/rsyslog 收集到 Kafka/ELK/S3，再本地按需拉取。
4. 如果必须 Python 从本地拉：使用 `asyncio` 或受控线程池，配合信号量限制并发，比如每批 50-200 台。
5. 每台机器只读取增量日志：按时间、文件 offset、日志滚动文件名记录 checkpoint。
6. 远端先过滤和压缩：只拉错误级别、目标时间段、目标模块日志，避免传输无用数据。
7. 本地写入要流式落盘，使用队列削峰，避免内存堆积。
8. 加入失败重试、超时、校验、失败清单，任务可恢复。

**结合个人项目的回答素材：**

- 你做过测试平台里的日志错误检查，简历中写有 DRF+Vue 测试平台涵盖接口自动化、UI 自动化、性能测试、日志错误检查。
- 你当前 Linux 能力是薄弱项，面试中更应该把架构思路说稳，不要陷入具体命令细节。
- 可以联系 AI+接口自动化 SKILL 的闭环思想：采集、解析、生成报告、失败重试、结果追踪，而不是一次性暴力执行。

**建议回答结构：**

```text
这个场景我不会让本地电脑一次性连 10000 台机器全量拉日志，因为本地连接数、内存、磁盘和网络都会扛不住。
我会先做架构拆分：如果公司已有日志体系，优先用 Filebeat、Fluent Bit、rsyslog 这类 agent 在远端采集，写入 Kafka、ELK 或对象存储，本地只按条件查询和下载。
如果必须用 Python 从本地主动采集，我会做分批限流和流式处理。比如用 asyncio 或线程池，每次只并发 50 到 200 台，通过信号量控制连接数；远端按时间范围和关键字过滤，再压缩传输；本地边读边写文件，不把日志一次性加载到内存。
同时我会记录每台机器的 offset 或最后采集时间，支持断点续传；对失败机器记录失败清单，设置超时、重试和校验。这样即使本地资源有限，也可以稳定地分批完成采集。
```

**避坑提醒：**

- 不要回答“开 10000 个线程去拉”。
- 不要把所有日志读入内存后再处理。
- 面试官可能继续问稳定性，要主动说 checkpoint、失败重试、压缩、过滤、限流。

### 问题5：Prompt 工程和 Harness 工程有什么区别？在你的项目中如何体现 Harness 工程？

**考察点：** AI 工程理解深度、简历中 harness engineering 的真实性、AI 项目质量控制能力。

**回答引导：**

1. Prompt 工程优化的是模型输入指令，重点是让单次回答更好。
2. Harness 工程优化的是 Agent 的运行环境，重点是约束、上下文、工具、反馈闭环和持续改进。
3. 可以用三个层级说明：Prompt 工程是“怎么说”，上下文工程是“给什么资料”，Harness 工程是“让 AI 在什么规则和工具链里稳定执行”。
4. 结合你的项目：前置门禁、渐进式披露、接口索引、抓包脚本、pytest 强制闭环、失败禁止跳过、人工确认断言。
5. 强调 Harness 不削弱 AI，而是给 AI 加边界，让它在正确范围内高效执行。

**结合个人项目的回答素材：**

- Obsidian 的 Harness Engineering 笔记定义：围绕 AI 智能体设计约束机制、反馈回路、工作流控制和持续改进循环，核心是“Human Steer, Agent Execute”。
- SKILL 发布说明明确写到：前置门禁、渐进式披露、pytest 闭环、避免 vibe coding 的随机性。
- 你的接口 SKILL 要求 AI 先填写接口方法/用例位置，再读取对应子文档，再执行最小范围 pytest，这就是 Harness 的体现。

**建议回答结构：**

```text
Prompt 工程主要优化单次输入，比如怎么写提示词让模型回答更准确、更结构化。
Harness 工程关注的是模型运行环境，它不只是一个提示词，而是把上下文、工具、规则、门禁、反馈循环和验收标准组合起来，让 Agent 能稳定完成任务。
在我的 AI+接口自动化 SKILL 里，Harness 工程体现得比较明显。比如我设计了前置门禁，要求 AI 必须先确认新增接口方法和用例写入位置，避免乱改文件；把不同工作流拆成子文档，按需读取，减少上下文过长；生成用例后必须执行最小范围 pytest，失败不能跳过，必须分析日志并修复；最后还要检查 Git diff 和人工确认业务断言。
所以 Prompt 解决的是“让 AI 这次答得好”，Harness 解决的是“让 AI 在项目里长期、稳定、可控地工作”。
```

**避坑提醒：**

- 不要把 Harness 简单解释成“高级提示词”。
- 要讲你项目里的具体护栏，否则面试官会怀疑简历关键词堆砌。
- 可以主动说 AI 仍需人工掌舵，特别是业务 BUG 和用例维护边界判断。

### 问题6：怎么保证需求文档到用例输出的正确性？

**建议回答结构：**

```text
我会从需求解析、覆盖追踪、用例设计方法、输出校验和执行验证几个层面保证正确性。
第一步先把需求拆成可测试点，包括角色、权限、输入输出、状态流转、异常规则和验收标准。第二步建立需求点和用例的映射，保证每个需求至少有正常场景、异常场景、边界场景和权限场景。

对于接口自动化用例，我还会执行 pytest，通过真实接口返回、数据库或关键字段断言来验证用例是否能落地。这样可以避免用例只是在文档上看起来完整，但实际不可执行。
```

**避坑提醒：**

- 不要说“AI 生成后人工看一下”就结束，要讲追踪矩阵和执行验证。

### 问题7：向量入库存入过程，以及怎么在框架里添加向量数据库？

**考察点：** RAG 基础、Embedding 流程、向量数据库选型、测试框架扩展能力。

**回答引导：**

1. 先讲向量入库流程：文档加载、清洗、切分 chunk、加 metadata、调用 embedding 模型、生成向量、upsert 到向量库、建立索引。
2. 查询流程：用户问题向量化，向量库 Top-K 检索，必要时加 BM25 混合检索和 rerank，再把上下文交给 LLM。
3. metadata 很重要：来源文件、模块、接口、版本、权限、时间、项目名，便于过滤和溯源。
4. 在测试框架中添加向量库：先选型，轻量用 Chroma/FAISS，生产大规模用 Milvus。
5. 增加 ingestion 脚本：扫描需求文档、接口文档、历史用例、失败日志，定期入库。
6. 增加 retrieval 服务：根据需求或接口名检索相似历史用例、规范和断言模板。
7. 增加评估：Precision@K、Recall@K、Hit Rate、Answer Relevancy、Faithfulness。

**结合个人项目的回答素材：**

- 你 Obsidian RAG 笔记中明确记录了索引、检索、生成，以及 BM25、语义搜索、混合搜索、元数据过滤、重排序、Precision@K、Recall@K。
- LangChain 笔记中记录了 FAISS、ChromaDB、Milvus、Pinecone 等向量存储选型。
- 你的接口 SKILL 当前已有接口索引 SQLite，可自然扩展为“结构化索引 + 向量索引”：SQLite 查精确接口，向量库查相似用例/需求说明。

**建议回答结构：**

```text
向量入库一般分为加载、切分、向量化和入库几个步骤。
先读取需求文档、接口文档、历史用例或日志，清洗无效内容，再按标题、段落、接口、用例这种结构切成 chunk。每个 chunk 都要带 metadata，比如来源文件、模块、接口名、版本、权限和更新时间。然后调用 embedding 模型把文本转成向量，最后把向量、原文和 metadata 一起 upsert 到向量数据库。
查询时，把用户问题或需求点也转成向量，到向量库里做 Top-K 相似检索，必要时结合 BM25 做混合检索，再用 rerank 重排，最后把最相关的上下文给大模型生成用例。
如果加到我的接口自动化框架里，我会保留现有 SQLite 接口索引用来查精确 URL 和方法位置，再增加一个向量库模块，比如本地先用 Chroma 或 FAISS，存历史用例、接口说明、需求文档和失败案例。AI 生成用例前先检索相似用例和业务规则，再生成和执行 pytest，这样比单纯靠 Prompt 更可靠。
```

**避坑提醒：**

- 不要只说“把文本转成向量存进去”，要讲 chunk 和 metadata。
- 不要忽略权限过滤，企业 RAG 里权限和租户隔离很重要。
- 可以主动说本地原型用 Chroma/FAISS，大规模生产可考虑 Milvus。

### 问题8：中间件怎么测？

**考察点：** 测试广度、集成测试、性能测试、稳定性测试、故障注入和监控意识。

**回答引导：**

1. 先说明中间件不是只测功能，要测功能、集成、性能、稳定性、异常恢复和监控。
2. 按类型举例：Redis、MQ、Nginx、Tomcat、数据库连接池、缓存、网关。
3. Redis：缓存读写、过期、淘汰、穿透/击穿、主从切换、内存、命中率。
4. MQ：消息发送、消费、顺序、重复、重试、死信队列、积压、幂等。
5. Nginx/网关：路由转发、负载均衡、限流、超时、502/504、HTTPS。
6. Tomcat/应用容器：线程池、连接数、超时、JVM、GC、日志。
7. 性能和监控：TPS、响应时间、错误率、CPU、内存、队列深度、连接池使用率。

**结合个人项目的回答素材：**

- 性能测试基础笔记中记录了中间件监控：Tomcat 线程池、Redis 内存、MQ 队列深度、Nginx 连接数。
- 你简历中有 JMeter、Locust、性能测试和日志错误检查，可以把中间件测试和性能监控结合。
- 泛微 E10 是分布式微服务架构，回答中可以说测试时会结合接口链路、日志和依赖组件状态定位问题。

**建议回答结构：**

```text
中间件测试我会从功能、集成、性能、稳定性和异常恢复几个方面看。
比如 Redis，我会测缓存读写、过期时间、缓存未命中、缓存穿透和内存使用；MQ 会测消息发送消费、顺序性、重复消费、重试、死信队列和消息积压；Nginx 或网关会测路由转发、负载均衡、限流、超时、502/504；Tomcat 这类应用容器会关注线程池、连接数、超时和 JVM 指标。
在性能测试里，我会结合 JMeter 或 Locust 压接口，同时监控中间件指标，比如 Redis 内存和命中率、MQ 队列深度、Nginx 连接数、Tomcat 线程池、数据库连接池。发现接口慢或错误率上升时，再结合日志和监控判断瓶颈是在应用、数据库还是中间件。
```

**避坑提醒：**

- 不要只说“看中间件能不能启动”。
- 中间件测试最好结合具体组件举例。
- 面试官追问性能瓶颈时，要能说出监控指标。

### 问题9：怎么保证这么多项目执行的稳定性？

**考察点：** 自动化测试工程能力、CI/CD、环境隔离、失败分析、测试平台运维意识。

**回答引导：**

1. 先说稳定性来自流程和工程化，不是靠人盯。
2. 项目隔离：环境隔离、配置隔离、数据隔离、依赖版本固定。
3. 用例稳定：分层、标签、优先级、最小可执行集、前后置清理、fixture 管理。
4. 执行稳定：Jenkins 定时/流水线、失败重跑只针对环境类失败，不能掩盖功能 BUG。
5. 资源控制：并发限制、任务队列、超时、锁、浏览器实例回收。
6. 报告和日志：统一报告、失败截图、请求响应、服务日志、失败分类。
7. 质量闭环：失败原因归类为功能问题、环境问题、脚本问题、数据问题，定期治理不稳定用例。
8. AI 项目中还要加规则约束：AI 不能跳过失败用例，必须执行 pytest 闭环。

**结合个人项目的回答素材：**

- 你的测试平台覆盖接口自动化、UI 自动化、性能测试、日志错误检查，并通过 Jenkins 集成。
- AI+接口自动化 SKILL 已经有强制 pytest 闭环、禁止跳过失败、Git diff 检查、前置门禁。
- Obsidian 中记录过 DeepSeek 误把失败用例跳过的问题，后续通过规则约束解决，这是一个很好的稳定性案例。

**建议回答结构：**

```text
我理解自动化项目多了以后，稳定性主要靠工程化治理。
第一是环境和数据隔离，不同项目的配置、账号、测试数据、依赖版本要分开，避免互相污染。第二是用例本身要稳定，通过 pytest fixture 做前置和清理，用标签区分冒烟、回归、专项用例，优先保证核心链路稳定。
第三是执行层要有任务队列、并发限制、超时控制和资源回收，UI 自动化还要保证浏览器实例关闭，接口自动化要避免测试数据重复导致失败。第四是报告和日志要完整，失败时能看到请求、响应、截图、日志和执行环境。
在我的 AI+接口自动化 SKILL 里，我还加了一个规则：AI 生成或维护用例后必须执行最小范围 pytest，失败不能跳过，必须分析原因。以前遇到过模型为了让结果通过而跳过失败用例的问题，后来通过规则约束和检查项把这个行为禁止掉。
```

**避坑提醒：**

- 不要把“失败重跑”当成唯一稳定性方案。
- 失败重跑只能处理环境抖动，不能掩盖真实 BUG。
- 要主动讲失败分类和长期治理，否则容易显得只是会跑脚本。

### 问题10：Linux 复制文件与复制目录的命令是什么？

**考察点：** Linux 基础命令、日常服务器操作能力。

**回答引导：**

1. 复制文件：`cp 源文件 目标文件`。
2. 复制目录：`cp -r 源目录 目标目录`，`-r` 表示递归复制目录下内容。
3. 保留文件属性：`cp -a 源目录 目标目录`。
4. 覆盖确认：`cp -i`，强制覆盖：`cp -f`。
5. 远程复制可补充：`scp` 或 `rsync -avz`，但先回答本地 `cp`。

**结合个人项目的回答素材：**

- 个人能力评估中 Linux 命令是当前薄弱项，真实面试已问到最基础命令，必须优先背熟。
- Obsidian Linux 总结中记录了 `cp [-rif] 源文件 目标文件`，其中 `-r` 用于复制文件夹和文件夹下所有文件。

**建议回答结构：**

```text
复制文件用 cp，比如：
cp a.log /tmp/a.log

复制目录要加 -r，表示递归复制目录下的所有内容，比如：
cp -r logs /tmp/logs

如果要尽量保留权限、时间等属性，可以用：
cp -a logs /tmp/logs
```

**避坑提醒：**

- 复制目录一定要加 `-r` 或 `-a`。
- 不要把复制命令和移动命令 `mv` 混淆。
- 这类基础题要回答短、准、稳。
