## 二、项目一句话定位

```text
测试平台是把接口自动化、UI自动化、性能测试、日志错误检查、任务调度、报告展示和环境配置集中管理的平台化系统，目标是让自动化测试从个人脚本升级为团队可用的工程能力。
```

结合我的项目说法：

```text
我参与的测试平台主要服务泛微E10低代码平台，后端用DRF提供接口，前端用Vue承载项目、用例、任务和报告页面，执行层对接接口自动化、UI自动化、性能测试、日志错误检查和Jenkins。它的价值是把原来分散在脚本、Excel、Jenkins任务和本地报告里的自动化能力统一管理，让团队能追溯每次执行的项目、环境、用例、构建、报告和失败原因。
```

## 三、平台核心模块

| 模块 | 作用 |
|------|------|
| 项目管理 | 管理不同业务项目和模块 |
| 环境管理 | dev/test/staging环境配置、base_url、账号 |
| 用例管理 | 接口/UI/性能用例元数据 |
| 任务管理 | 选择用例、环境、分支、触发执行 |
| 执行调度 | 调用pytest、Playwright、JMeter/Locust或Jenkins |
| 报告管理 | 展示通过率、失败原因、日志、截图 |
| 日志错误检查 | 收集服务日志、过滤ERROR/Exception |
| 权限管理 | 用户、角色、项目权限 |
| 通知管理 | 企业微信/邮件/平台消息 |
| AI能力 | AI生成用例、失败分析、报告总结 |

Obsidian测试平台基础中的核心主线可以这样背：

```text
项目管理解决“测哪个系统”，用例管理解决“测什么”，测试执行解决“什么时候、在哪个环境、由谁触发”，报告统计解决“结果如何、趋势如何、问题在哪里”。
```

结合E10低代码平台：

| 能力 | 在E10里的落点 |
|------|--------------|
| 接口自动化 | 表单、页面、报表、应用、菜单、数据仓库等模块接口回归 |
| UI自动化 | 低代码平台核心页面流程、配置流程、关键业务路径 |
| 性能测试 | 对核心接口或关键流程做JMeter/Locust压测和趋势观察 |
| 日志错误检查 | 自动化执行后过滤服务日志中的ERROR、Exception、Timeout |
| Jenkins集成 | 平台传分支、环境、用例标签，由Jenkins执行并归档报告 |

## 四、整体架构

```text
前端 Vue
  - 项目/用例/任务/报告页面
  - 触发任务、查看状态、展示报告

后端 DRF
  - REST API
  - 权限、任务、报告、环境、日志接口

执行层
  - pytest 接口自动化
  - Playwright/Selenium UI自动化
  - JMeter/Locust 性能测试
  - Jenkins Pipeline 调度

数据层
  - MySQL 存项目、用例、任务、报告、日志索引
  - 文件/对象存储 保存报告、截图、日志
```

更贴近面试的四层表达：

```text
前端层：Vue页面，负责项目、用例、执行、报告统计的配置和展示。
后端层：DRF接口，负责权限、数据校验、任务创建、报告查询和执行状态流转。
执行层：pytest、Playwright、JMeter/Locust、日志检查脚本和Jenkins Pipeline。
数据层：MySQL保存结构化数据，报告、截图、trace、日志作为文件或对象存储关联到执行记录。
```

## 五、前端架构怎么回答

真实面试问过“你们前端架构是什么样子”。

稳妥回答：

```text
我们测试平台整体是前后端分离架构，后端用Django REST Framework提供接口，前端用Vue做页面和交互。前端大概分为路由层、页面层、组件层、API请求层和状态管理层。页面包括项目管理、用例管理、任务执行、报告展示、日志检查等；组件层封装表格、表单、弹窗、报告图表；API层统一调用后端接口。真正的自动化执行由后端或Jenkins调pytest、Playwright等完成，前端负责配置、触发和展示。
```

注意：
- 不要把自己包装成专职前端架构师。
- 强调你熟悉测试平台业务和前后端协作。

## 六、DRF后端负责什么

DRF后端职责：
- 提供REST API。
- 处理用户认证和权限。
- 管理项目、用例、任务、报告。
- 触发Jenkins或本地执行器。
- 接收执行结果。
- 提供报告查询接口。
- 记录操作日志。

DRF常用能力：
- Serializer：序列化、反序列化、数据校验。
- ViewSet/APIView：接口逻辑。
- Router：路由。
- Permission：权限控制。
- Pagination/Filter：分页过滤。

你之前DRF薄弱点：
```text
Serializer不只是序列化和反序列化，还负责数据验证。
```

## 七、用例、任务、报告、环境怎么建模

### 1. 项目表

字段：
- id
- name
- owner
- repo_url
- description

### 2. 环境表

字段：
- id
- project_id
- env_name
- base_url
- db_config
- account_config

### 3. 用例表

字段：
- id
- project_id
- case_type
- case_name
- path
- marker
- priority
- status
- automation_type：interface/ui/performance/log_check
- module_name：E10业务模块，如表单、页面、报表、应用
- jenkins_marker：传给Jenkins或pytest的标签

### 4. 任务表

字段：
- id
- project_id
- env_id
- branch
- trigger_user
- status
- jenkins_build_id
- start_time
- end_time

### 5. 报告表

字段：
- id
- task_id
- total
- passed
- failed
- skipped
- pass_rate
- report_url
- log_url

面试表达：
```text
测试平台建模的核心是形成项目、环境、用例、任务、报告之间的追溯链路。一个任务要知道在哪个项目、哪个环境、哪个分支、执行了哪些用例、结果报告在哪里、失败日志在哪里。
```

结合Obsidian平台基础的追溯关系：

```text
项目(project) -> 用例(test_case) -> 执行(test_execution/task) -> 报告(report/statistics)
```

扩展说明：

```text
如果是接口和UI自动化，用例表可以记录脚本路径、pytest marker、所属模块；如果是性能测试，用例表记录压测脚本、并发配置和指标阈值；如果是日志错误检查，用例表记录服务名、日志路径、关键字和排除规则。这样不同测试类型都能通过统一的任务和报告模型进行管理。
```

## 八、如何触发自动化执行

方式：
1. 平台直接调用本地执行器。
2. 平台调用Jenkins API触发构建。
3. Jenkins拉代码执行pytest/Playwright。
4. 执行结果归档并回写平台。

推荐表达：

```text
测试平台更适合作为调度和展示入口，真正执行交给Jenkins或执行节点。平台传项目、分支、环境、用例标签等参数给Jenkins，Jenkins执行后把报告地址、通过率、失败详情回写平台。
```

## 九、如何和Jenkins集成

流程：

```text
平台创建任务 -> 调Jenkins buildWithParameters -> Jenkins拉代码 -> 执行pytest -> 生成报告 -> 平台轮询/回调获取结果 -> 更新任务状态
```

参数：
- branch
- env
- markers
- test_path
- build_user

失败排查：
- Jenkins Console Output。
- pytest报告。
- Allure/HTML报告。
- 请求响应日志。
- 服务端日志。
- Docker容器日志。

E10项目可背口径：

```text
在E10低代码平台里，平台更像调度入口，Jenkins更像执行入口。平台创建任务时传入项目、分支、环境、用例类型和marker，Jenkins拉取对应代码后执行接口/UI/性能或日志检查任务，执行完成后把报告地址、通过率、失败用例、日志链接回写平台。这样测试人员不用直接登录Jenkins找任务，也能在平台上看到每次回归的完整链路。
```

## 十一、结合AI能力

AI可以增强测试平台：
- AI生成接口用例。
- AI分析pytest失败日志。
- AI总结报告。
- AI推荐失败分类。
- AI从需求生成测试点。
- AI检索历史相似问题。

但要有边界：
```text
AI建议可以作为辅助，任务状态、缺陷定级、业务断言和上线结论仍需要人工确认。
```

和AI接口自动化SKILL系统的结合：

```text
测试平台负责管理项目、用例、任务和报告；AI+SKILL系统负责提高接口用例生成和维护效率。两者结合时，平台可以作为入口选择项目、模块、环境和接口来源，AI生成候选用例后仍然进入pytest执行、失败修复、报告回写和人工确认流程。这样AI能力不是脱离平台单独存在，而是嵌入自动化资产管理闭环。
```

## 十三、回答模板

```text
我理解测试平台的价值是把自动化能力平台化。整体上是DRF+Vue前后端分离，前端负责项目、用例、任务和报告页面，后端负责REST API、权限、任务、报告、环境配置和执行调度。真正的自动化执行可以交给pytest、Playwright、JMeter/Locust或Jenkins。数据模型上要形成项目、环境、用例、任务、报告的追溯链路，知道哪个分支、哪个环境、哪些用例、哪个构建产生了什么报告。和Jenkins集成时，平台通过参数触发构建，Jenkins执行测试并归档报告，结果再回写平台。AI能力可以接入用例生成、失败分析和报告总结，但关键业务判断仍要人工确认。
```

```text
T：我的任务是参与测试平台建设，把自动化测试能力平台化，让测试人员能统一管理项目、用例、任务和报告。
A：我基于DRF+Vue参与后端接口和平台能力建设，围绕项目、环境、用例、任务、报告建模；执行层对接pytest、Playwright、JMeter/Locust和Jenkins；报告侧沉淀通过率、失败原因、日志、截图和构建链接。
R：平台让接口/UI/性能/日志错误检查可以统一调度和追踪，也为AI生成用例、失败分析和报告总结提供了统一入口。
```
