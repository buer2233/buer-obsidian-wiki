## 二、UI自动化定位

UI自动化不是把所有功能都点一遍，而是覆盖：
- 核心业务链路。
- 高频回归场景。
- 端到端集成流程。
- 跨浏览器/关键兼容场景。
- 用户体验层面的关键状态。

不适合：
- 频繁变化的一次性页面。
- 纯视觉细节。
- 复杂但可由接口层低成本覆盖的规则。
- 成本高于收益的低频场景。

面试表达：
```text
我不会追求UI自动化全量覆盖，核心业务规则优先放接口层，UI层覆盖用户最关心的关键路径，比如登录、表单提交、审批流、报表查看、核心配置保存。
```

## 三、Selenium和Playwright区别

| 对比 | Selenium | Playwright |
|------|----------|------------|
| 协议/机制 | WebDriver生态成熟 | 现代浏览器自动化 |
| 等待 | 更多依赖显式等待 | 内置自动等待和可重试断言 |
| 定位 | By.ID/XPath/CSS等 | Locator，语义化定位更强 |
| 上下文隔离 | 需自行管理 | BrowserContext隔离方便 |
| 调试 | 传统日志截图 | trace、video、screenshot体验更好 |
| 适用 | 老项目、企业已有框架 | 现代前端、异步页面、稳定E2E |

面试模板：
```text
Selenium生态成熟，兼容性好，适合传统企业项目；Playwright更现代，Locator、自动等待、可重试断言和上下文隔离更强，处理异步页面更稳定。我会根据项目现状选择，不会简单说谁完全替代谁。
```

## 四、Playwright自动等待

Playwright在执行动作前会做 actionability checks。常见检查包括：
- 元素是否可见。
- 元素是否稳定。
- 元素是否可以接收事件。
- 元素是否启用。
- 对部分输入动作，还会检查是否可编辑。

你之前漏过 receives events，要重点背：
```text
visible、stable、receives events、enabled，输入类还要 editable。
```

示例：

```python
page.get_by_role("button", name="Submit").click()
```

上面 click 前，Playwright会自动等待按钮满足可操作条件。

## 五、什么时候还需要显式等待

自动等待不是万能。以下场景仍需要显式等待：

| 场景 | 示例 |
|------|------|
| 业务状态变化 | 提交后等待“保存成功” |
| 异步任务完成 | 报表生成完成 |
| 下载/上传事件 | expect_download |
| 接口响应完成 | 等特定 response |
| 自定义条件 | 表格行数变化、状态从处理中变为完成 |

Playwright示例：

```python
from playwright.sync_api import expect

page.get_by_role("button", name="保存").click()
expect(page.get_by_text("保存成功")).to_be_visible()
```

下载示例：

```python
with page.expect_download() as download_info:
    page.get_by_text("下载").click()
download = download_info.value
download.save_as("report.xlsx")
```

## 六、Selenium等待

| 等待 | 特点 | 建议 |
|------|------|------|
| 隐式等待 | 全局查找元素等待 | 少用，避免和显式等待混用 |
| 显式等待 | 等具体条件 | 推荐 |
| Fluent Wait | 自定义轮询间隔和忽略异常 | 复杂场景 |

示例：

```python
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC
from selenium.webdriver.common.by import By

WebDriverWait(driver, 10).until(
    EC.element_to_be_clickable((By.ID, "submit"))
).click()
```

面试表达：
```text
等待策略的核心是等待业务状态，不是等待固定时间。Selenium里我以显式等待为主，Playwright里优先使用Locator和expect。
```

## 七、Locator优势

Playwright Locator优势：
- 延迟定位。
- 自动等待。
- 可重试。
- 可组合。
- 推荐语义化定位，如 role、text、label、test id。

示例：

```python
page.get_by_role("button", name="提交")
page.get_by_label("用户名").fill("dwp")
page.get_by_test_id("order-submit").click()
```

## 八、PO模式

PO模式目标：把页面操作细节从测试用例中抽离。

```python
class LoginPage:
    def __init__(self, page):
        self.page = page
        self.username = page.get_by_label("用户名")
        self.password = page.get_by_label("密码")
        self.submit = page.get_by_role("button", name="登录")

    def login(self, username, password):
        self.username.fill(username)
        self.password.fill(password)
        self.submit.click()
```

大型项目建议进一步拆组件：
- TableComponent
- FormComponent
- DialogComponent
- MenuComponent

面试表达：
```text
低代码平台页面结构变化多，我会在PO基础上做组件封装，比如表格、弹窗、表单组件。用例层只描述业务流程，定位和等待细节集中在页面对象和组件对象里。
```

结合我的真实项目：
```text
在国科智慧社区项目里，我用Python+Selenium+pytest搭过PO模式框架，主要覆盖Web端登录、物业缴费、报修工单、消息通知等核心流程。那类系统页面相对固定，所以PO层可以按页面拆，比如登录页、报修页、缴费页。

到泛微E10低代码平台后，页面、表单、流程是配置生成的，单纯按页面拆PO维护成本会变高。所以我会在PO基础上再抽组件层，比如表单组件、表格组件、弹窗组件、流程提交组件。用例层只表达“创建表单-填写字段-提交流程-校验结果”，具体定位、等待、弹窗处理放到组件对象里。
```

面试被追问“你这个PO框架到底怎么分层”时，可以这样答：
```text
我一般分四层：第一层是基础Driver或Page封装，统一浏览器、等待、截图、日志；第二层是公共组件，比如表单、表格、弹窗、菜单；第三层是业务页面对象，比如低代码表单设计页、流程审批页；第四层才是pytest测试用例。这样做的好处是页面元素变化时，多数只改页面对象或组件对象，不会影响大量用例。
```

## 十、UI自动化不稳定怎么解决

按顺序排查：
1. 定位是否稳定，避免过长XPath。
2. 等待是否等待了业务状态。
3. 测试数据是否独立。
4. 环境是否稳定。
5. 页面是否有动画、遮罩、异步加载。
6. 失败时是否有截图、trace、video、日志。
7. 是否需要重试，但不能用重试掩盖真实BUG。

结合E10低代码平台的排查口径：
```text
低代码页面常见不稳定点不是脚本本身，而是页面异步渲染、字段联动、弹窗遮罩、流程状态刷新、表格数据延迟加载。我的处理思路是先把sleep换成业务状态等待，比如等待“保存成功”、流程状态变成“已提交”、表格出现目标记录；再检查定位是否依赖动态ID；最后通过截图、日志、失败页面HTML或trace定位原因。
```

面试被追问“你怎么证明不是脚本不稳定，而是真BUG”时，可以这样答：
```text
我会看三类证据：第一是失败时截图或trace，确认页面当时状态；第二是前后端日志或接口响应，判断保存、提交、查询是否真的失败；第三是手工复现同一数据和步骤。如果自动化失败点能手工复现，并且接口或页面状态也异常，我才会按BUG提交；如果只是环境慢或定位变化，就归为脚本维护问题。
```

## 十一、结合AI+UI自动化/browser-use

面试表达：
```text
browser-use类项目不是替代Playwright，而是在Playwright底层浏览器控制能力上增加LLM任务理解、页面观察、动作规划和失败重试。传统Playwright适合确定性流程，AI+UI自动化适合自然语言任务、页面变化较多和探索性流程。但工程上仍要记录每一步动作、截图、失败原因和最终断言，不能只看AI说完成了。
```

结合我的真实项目/经验：
```text
我的UI自动化经验分两段：早期在国科智慧社区，用Selenium+pytest+PO模式解决Web/APP后台核心流程回归；现在在泛微E10低代码平台，既做传统UI自动化，也尝试AI+UI自动化。传统脚本适合稳定回归，比如登录、表单保存、流程提交；browser-use这类AI浏览器自动化更适合低代码平台中页面变化多、操作路径不固定的探索性场景。
```

## 十三、回答模板

```text
我做UI自动化会优先覆盖核心链路，而不是全量堆脚本。稳定性主要靠稳定定位、合理等待、数据隔离和失败可观测。Playwright相比Selenium更现代，Locator、自动等待、可重试断言和上下文隔离更强；Selenium生态成熟，适合已有企业框架。等待上，我会等业务状态而不是sleep，Playwright优先locator和expect，Selenium优先WebDriverWait。对于低代码平台，我会用PO加组件封装降低维护成本。AI+browser-use项目则是在Playwright底层能力上增加任务理解和动作规划，但关键断言和风险控制仍要工程化处理。
```

结合我简历的1分钟面试回答：
```text
我有两类UI自动化经验。第一类是国科智慧社区，当时用Python+Selenium+pytest搭过PO模式框架，覆盖Web端和APP相关后台流程，比如登录、物业缴费、报修工单、消息通知。第二类是泛微E10低代码平台，我负责页面、表单、流程类场景的自动化，低代码页面变化比较多，所以我不会只按页面写死PO，而是会抽表单、表格、弹窗、流程提交这些公共组件。

如果项目是老系统或已有Selenium框架，我会继续沿用Selenium并做好显式等待和PO封装；如果是现代前端或异步加载多的页面，我更倾向Playwright，因为它有Locator、自动等待、可重试断言和BrowserContext隔离。我的核心原则是：UI层只覆盖关键业务链路，业务规则尽量下沉到接口层，避免UI自动化变成高成本维护。
```

面试被追问“Playwright自动等待到底等什么”时，可以这样答：
```text
Playwright点击前会做可操作性检查，我会按visible、stable、receives events、enabled来记，输入类还会检查editable。这里receives events容易漏，它的意思是元素不能被遮罩挡住，能真正接收到点击事件。自动等待解决的是元素可操作问题，但业务完成仍要用expect等待业务状态，比如保存成功提示、流程状态变化、表格出现目标数据。
```
