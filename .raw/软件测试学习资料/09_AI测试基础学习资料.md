## 二、AI测试和传统测试区别

| 维度 | 传统系统 | AI系统 |
|------|----------|--------|
| 输出 | 多数确定 | 概率性、语义性 |
| 断言 | 精确匹配较多 | 需要语义评分和阈值 |
| 缺陷 | 功能错误、兼容性 | 幻觉、偏见、不稳定、不安全 |
| 回归 | 输入输出固定 | 需要评估集和多次运行 |
| 监控 | 错误率、性能 | 还要监控幻觉率、拒答率、成本 |

面试表达：
```text
AI测试最大区别是输出不完全确定，不能只靠一次精确断言，而要通过评估集、指标、重复运行、人工抽检和线上反馈来判断是否满足业务目标。
```

## 三、AI应用测试维度

| 维度 | 测试点 |
|------|--------|
| 功能 | 是否完成任务 |
| 准确性 | 回答是否符合事实和业务规则 |
| 相关性 | 是否回答用户问题 |
| 忠实性 | 是否基于给定上下文 |
| 幻觉 | 是否编造事实 |
| 安全 | prompt injection、敏感信息、越权 |
| 鲁棒性 | 错别字、歧义、多轮、边界 |
| 性能 | 响应时间、P95、并发 |
| 成本 | token、模型调用费用 |
| 可观测性 | 日志、trace、评分、反馈 |

面试展开口诀：**业务可用、答案可信、过程可控、风险可防、成本可算、问题可追**。

| 展开维度 | 面试时要说到的细节 |
|----------|--------------------|
| 业务可用 | 是否完成用户任务，是否符合业务规则，失败时是否追问或转人工 |
| 答案可信 | 是否基于事实/上下文，是否有幻觉，引用是否能支撑答案 |
| 过程可控 | Prompt、RAG、工具调用、后处理是否有日志和可回放链路 |
| 风险可防 | Prompt Injection、越权、敏感信息、危险建议是否被拦截 |
| 成本可算 | token、模型调用次数、P95延迟、失败重试成本是否可接受 |
| 问题可追 | 每次输入、上下文、模型版本、评分、人工反馈是否能沉淀 |

## 四、评测集设计

AI测试的核心资产是评测集。

评测集应包含：
- 高频真实用户问题。
- 正常问题。
- 边界问题。
- 无答案问题。
- 安全攻击问题。
- 多轮上下文问题。
- 业务规则冲突问题。
- 历史失败样本。

字段示例：

```json
{
  "id": "case_001",
  "question": "如何创建订单？",
  "expected_answer": "需要选择商品、填写地址并提交订单",
  "expected_behavior": "answer",
  "risk_level": "medium",
  "tags": ["order", "normal"]
}
```

## 五、自动化评分方法

| 方法 | 适用 | 风险 |
|------|------|------|
| 规则评分 | 格式、关键词、拒答 | 覆盖有限 |
| 精确匹配 | 固定输出 | 不适合开放回答 |
| 语义相似度 | 问答质量 | 可能误判 |
| LLM-as-judge | 开放问题评分 | 需人工校准 |
| 人工抽检 | 高风险样本 | 成本高 |

推荐组合：
```text
低风险规则自动判，高风险人工抽检，开放语义题用LLM-as-judge但要定期和人工校准。
```

## 六、核心指标

| 指标 | 含义 |
|------|------|
| 准确率 | 回答正确比例 |
| 通过率 | 满足测试标准比例 |
| 幻觉率 | 编造或不忠实比例 |
| 拒答正确率 | 应拒答时正确拒答 |
| 误拒率 | 应回答却拒答 |
| 一致性 | 同一问题多次回答稳定性 |
| 响应时间 | 平均、P95、P99 |
| 成本 | token和单次调用成本 |
| 人工接管率 | 产品需要人工处理比例 |

## 七、AI缺陷定位

分层定位：

```text
用户输入 -> Prompt -> 检索/RAG -> 工具调用 -> 模型 -> 后处理 -> 权限/业务规则 -> 输出展示
```

常见定位方式：
- 最终答案错，先看输入和Prompt。
- RAG问题先看检索结果是否正确。
- Agent问题看工具调用轨迹。
- 安全问题看是否绕过系统指令或权限。
- 性能问题看模型延迟、工具延迟、检索延迟。

## 八、结合个人项目

你的真实经验可这样表达：

```text
我在AI+接口自动化SKILL项目里，实际做的就是把不确定的AI输出纳入可验证流程。AI可以生成接口方法和pytest用例，但生成后必须执行最小范围pytest，失败要根据日志修复，关键业务断言要人工确认。这本质上就是AI测试里的评估闭环：输入标准化、输出可执行、结果可验证、失败可回流。
```

结合简历核心优势，可以再补一段：

```text
这个项目不是单纯调用大模型，而是结合了Prompt工程、上下文工程和harness/loop engineering。Prompt负责约束输出格式，上下文工程负责给模型接口索引、历史用例、项目规则，harness负责限制写入位置、运行pytest、检查Git diff，loop负责把失败日志回传给AI继续修复。最后用AI使用前后5周数据验证效果，接口用例编写效率提升93.43%，综合效率提升46.48%。
```

如果面试官追问AI+UI/browser-use，可这样接：

```text
AI+UI自动化里，我把browser-use理解成浏览器Agent测试。它通过LLM观察页面、规划动作，再调用Playwright点击、输入和跳转。测试时不能只看最后页面是否成功，还要看动作轨迹、元素识别、失败重试、截图日志和高风险动作控制。
```

## 十、回答模板

```text
AI测试不是简单验证模型会不会回答，而是围绕业务目标做系统评估。我会先定义成功标准，再构建覆盖正常、边界、异常和安全风险的评测集；然后批量调用模型，用规则评分、LLM-as-judge和人工抽检评估准确性、稳定性、幻觉率、响应时间和成本。上线后还要记录日志、用户反馈和失败样本，把失败样本回流到评测集，形成持续回归。结合我的AI+接口自动化项目，核心就是把AI生成结果变成可执行、可验证、可回归的工程流程。
```
