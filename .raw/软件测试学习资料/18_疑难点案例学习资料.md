## 三、案例1：AI生成用例质量不稳定

### 现象

AI生成接口自动化用例时，有时会漏掉pytest执行、断言较浅、或者不按项目目录写文件。

### 排查

分析生成过程发现：
- 一次性给AI的上下文过长。
- 低频规则和核心规则混在一起。
- 没有强制写入位置。
- 只要求“生成用例”，没有要求执行闭环。

### 根因

直接vibe coding随机性太强，AI缺少工程化约束。

### 解决

- 拆分SKILL文档，按工作流渐进式读取。
- 增加前置门禁：必须明确接口方法和用例写入位置。
- 固化输出模板。
- 生成后强制执行最小范围pytest。
- 检查Git diff，人工确认关键业务断言。

### 结果

AI从“生成代码草稿”变成“生成可执行用例并通过验证”的流程，用例编写效率提升约93%。

### 面试回答模板

```text
我遇到过AI生成用例质量不稳定的问题。早期直接让AI写接口用例时，它有时会漏掉pytest执行，或者把文件写到不合适的位置。排查后发现，本质是上下文和规则没有工程化，AI只是在做一次性代码生成。
后来我把流程改成SKILL系统：先要求明确写入位置，再按抓包、cURL、参考已有用例等不同工作流读取对应规则，生成后必须执行最小范围pytest，失败要根据日志修复，最后检查Git diff和人工确认业务断言。这样把AI输出纳入执行闭环，稳定性明显提升。
```

STAR可背版本：

```text
S：E10低代码平台接口数量多，人工写接口方法和pytest用例效率低，我希望用Claude Code/Codex提升产出。
T：我的任务是让AI不只是生成代码，而是能按公司接口自动化框架稳定生成可执行用例。
A：我设计了SKILLS规则和4种工作流：抓包驱动、参考已有、cURL手工、Java Controller源码；同时增加写入位置确认、最小范围pytest执行、失败修复、Git diff审查和人工确认业务断言。
R：AI使用后5周接口用例从198条提升到383条，用例编写效率提升约93%，接口方法+用例综合产出提升约46%。
```

## 四、案例2：AI维护用例时无法判断BUG还是用例过期

### 现象

pytest失败后，AI根据接口返回倾向于直接修改断言，让用例通过。

### 排查

检查发现：
- 接口返回值变了。
- 仅凭响应无法判断是需求变更、接口BUG还是用例断言过期。
- AI缺少业务预期上下文。

### 根因

自动化失败有三种可能：功能BUG、用例问题、环境/数据问题。AI不能只看报错就直接改断言。

### 解决

- 要求AI给出页面复测步骤。
- 对照需求和历史版本。
- 确认业务预期。
- 如果是BUG，提交缺陷；如果是用例过期，再维护断言。

### 结果

避免了把真实BUG改成“用例通过”。

### 面试回答模板

```text
我在AI维护接口用例时遇到过一个典型问题：pytest报错后，AI倾向于按新返回值修改断言，但我没有直接接受。因为自动化失败可能是功能BUG，也可能是用例过期。
我让AI先给出页面复测步骤和判断依据，再结合需求和历史版本确认业务预期。最后发现数据为0时接口应该返回0而不是空，所以这是功能BUG，不是用例维护。我提交缺陷给开发处理。
这个案例让我形成了一个规则：AI可以辅助分析失败和给出复测路径，但BUG和用例维护的最终判断必须结合业务预期、历史版本和人工验证。
```

## 五、案例3：DeepSeek误跳过失败用例

### 现象

模型执行pytest失败后，为了让结果看起来通过，尝试给失败用例加skip或绕过断言。

### 排查

发现规则里只要求“修复失败”，没有明确禁止通过跳过用例来制造通过结果。

### 根因

模型优化目标偏向“完成任务”，如果没有验收规则，可能采取不符合测试原则的捷径。

### 解决

- 明确禁止skip/xfail掩盖失败。
- pytest失败必须分析原因。
- 修复后重新执行。
- Git diff检查是否有不合理跳过。

### 面试回答模板

```text
有一次模型维护失败用例时，为了让pytest通过，尝试把失败用例跳过。这个问题很典型，说明AI会为了完成目标走捷径。
我后面在SKILL规则里明确禁止用skip或xfail掩盖失败，要求失败必须分析原因，区分功能BUG、环境问题、数据问题或用例问题，并且修复后重新执行pytest。同时我会检查Git diff，确认没有绕过断言。
这个案例体现了AI测试工具落地不能只靠提示词，还要有验收规则和质量门禁。
```

## 六、案例4：UI自动化脚本不稳定

### 现象

UI自动化偶发失败，有时元素找不到，有时点击无效。

### 排查

按层排查：
- 定位是否依赖脆弱XPath。
- 页面是否有异步加载。
- 是否存在遮罩、动画、弹窗。
- 数据是否被前置用例污染。
- 环境响应是否慢。
- 是否缺少截图和日志。

### 根因

UI脚本没有等待业务状态，只是等待固定时间或直接点击。

### 解决

- Selenium改用显式等待。
- Playwright使用Locator和expect。
- PO模式封装页面操作。
- 失败时保留截图、trace、日志。
- 核心流程尽量下沉到接口层，UI只覆盖关键路径。

### 面试回答模板

```text
UI自动化不稳定时，我不会先加sleep，而是先看失败截图、日志和页面状态。常见原因是定位不稳定、页面异步加载、遮罩动画、测试数据污染或环境慢。
我的处理方式是优先改稳定定位，Selenium用显式等待，Playwright用Locator和expect等待业务状态；同时用PO或组件封装减少页面变化影响。对于可以在接口层验证的规则，我会下沉到接口自动化，UI层只保留核心链路。
```

## 七、案例5：Linux日志和磁盘排查能力不足的补强案例

这个案例可以诚实表达学习改进，不要硬装。

### 现象

面试和日常排查中，Linux基础命令不够熟，尤其是 `tail`、`grep`、`df`、`du`。

### 解决

建立固定排查SOP：

```bash
tail -f app.log
grep -n "ERROR\|Exception\|Timeout" app.log
df -h
du -sh * | sort -rh | head
find /var/log -type f -size +100M
```

### 面试回答模板

```text
我之前更多精力在接口自动化和AI测试项目上，Linux命令属于需要补强的基础项。现在我把日志和磁盘排查整理成固定SOP：接口报错先tail -f看实时日志，再grep -n定位ERROR和Exception；磁盘报警先df -h看分区，再du逐层定位大目录，必要时用find找大文件。这个补强能直接服务自动化失败排查和Jenkins环境维护。
```

## 八、案例6：10000台机器日志采集

真实面试系统设计题。

### 回答要点

不能：
- 开10000个线程。
- 一次性把所有日志读入内存。
- 本地电脑直接全量拉取。

正确：
- 优先用日志系统：Filebeat/Fluent Bit/rsyslog -> Kafka/ELK/对象存储。
- 如果必须Python本地采集，使用分批、限流、流式处理、断点续传。
- 远端过滤、压缩。
- 本地边读边写。
- 失败清单、超时、重试。

### 面试回答模板

```text
这个场景我不会让本地电脑一次性连10000台机器全量拉日志，因为连接数、内存、磁盘和网络都会扛不住。优先方案是用Filebeat或Fluent Bit在远端采集到日志平台，本地按条件查询。
如果必须用Python主动采集，我会用asyncio或受控线程池，配合信号量限制并发，比如每批50到200台；远端按时间和关键字过滤并压缩；本地流式写文件，不把日志一次性加载到内存。同时记录每台机器的offset或最后采集时间，支持断点续传，对失败机器记录失败清单并重试。
```

### 回答框架

```text
环境隔离 -> 数据隔离 -> 用例分层 -> 执行调度 -> 失败可观测 -> 失败分类治理
```

### 面试模板


## 十、薄弱项追问纠偏模板

这一节专门用于面试追问时“答偏了以后拉回来”。原则是先承认边界，再给结构化答案，不要硬编。

### 1. Linux日志/磁盘追问

容易答错：
- `tail` 只说 `tail -f` 不带文件。
- `grep` 写成反引号或不会组合关键字。
- `du`、`df` 混淆。

纠偏模板：

```text
这个点我补充得更准确一点：Linux排查我会按日志和磁盘两条线走。日志先用tail -f app.log看实时输出，再用grep -n "ERROR\|Exception\|Timeout" app.log定位关键行；磁盘先用df -h看分区是否满，再用du -sh * | sort -rh | head逐层定位大目录。这个能力主要服务Jenkins执行失败、接口报错和服务日志排查。
```

### 2. Docker基础追问

容易答错：
- 说不清镜像、容器、仓库。
- docker-compose只会说“多服务部署”，说不出依赖和网络。

纠偏模板：

```text
Docker我按三个核心概念理解：镜像是只读模板，容器是镜像运行出来的实例，仓库是保存和分发镜像的地方。测试平台或自动化环境里，Docker主要解决环境一致性；docker-compose用于编排后端、前端、MySQL、Redis、执行节点等多服务，重点关注端口、环境变量、volume、depends_on和日志。
```

### 3. HTTP协议追问

容易答错：
- 401/403混淆。
- POST/PUT/PATCH混淆。

纠偏模板：

```text
我纠正一下：401是未认证，通常表示没有登录或token无效；403是已认证但没有权限。POST一般用于创建或提交动作，PUT偏整体更新，PATCH偏局部更新。在接口自动化里，我会重点验证状态码、业务code、权限场景、幂等性和错误响应，不只看请求是否返回200。
```

### 4. pytest Hook追问

容易答错：
- 只说“收集前、执行前、执行后”，说不出函数名。

纠偏模板：

```text
pytest Hook我现在会按生命周期记：pytest_configure用于初始化配置；pytest_collection_modifyitems可以在收集后修改用例或marker；pytest_runtest_makereport可以拿到setup/call/teardown阶段的执行结果，常用于失败截图、日志和报告增强；pytest_sessionstart和pytest_sessionfinish适合记录整个会话的开始和结束。我们真实项目里也用Hook增强pytest-html报告和执行结果记录。
```

### 5. browser-use Element Tree追问

容易答错：
- 只说DOM和视觉，不知道Element Tree压缩。

纠偏模板：

```text
browser-use不是把完整DOM原样丢给大模型，而是会把页面中可交互、有意义的元素抽取成更紧凑的Element Tree，例如按钮、输入框、链接、文本和它们的层级/属性。这样可以减少上下文噪音，让模型更容易基于当前页面状态决定下一步操作。我的二开重点是多LLM接入、执行流程适配和测试场景落地，但面试时我会补充Element Tree是它降低LLM理解成本的关键机制之一。
```

### 6. RAG指标追问

容易答错：
- 只说“准确性、召回率”，没有阈值和自动化方案。

纠偏模板：

```text
RAG测试我会拆成检索和生成两层。检索层看Recall@K、TopK命中率、MRR、无关文档比例；生成层看答案正确性、引用一致性、幻觉率、拒答能力和上下文忠实度。自动化上可以准备标准问题集、标准答案和期望引用文档，跑批后统计命中率、答案评分和失败样本，再人工抽查高风险问题。
```

### 7. Locust追问

容易答错：
- 只知道协程优势，不会讲脚本结构。

纠偏模板：

```text
Locust的优势是Python脚本化和协程并发，适合把接口自动化经验迁移到性能测试。脚本结构上通常定义HttpUser，wait_time控制用户等待时间，@task定义用户行为，self.client.get/post发请求。面试时如果不要求手写完整代码，我会重点讲清用户模型、任务权重、并发用户数、spawn rate、响应时间、TPS和失败率这些指标。
```
