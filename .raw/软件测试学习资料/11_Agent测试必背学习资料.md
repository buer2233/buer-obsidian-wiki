## 二、Agent是什么

Agent = LLM + 工具 + 状态/记忆 + 规划/决策 + 执行循环。

通用流程：

```text
接收任务 -> 理解目标 -> 拆分步骤 -> 选择工具 -> 调用工具 -> 观察结果 -> 调整计划 -> 输出结果
```

区别于普通LLM：
- 普通LLM主要输出文本。
- Agent会调用工具、执行动作、维护状态。
- 因此风险从“说错”变成“做错”。

---

## 三、Agent测试三层

| 层级 | 测什么 |
|------|--------|
| 结果层 | 最终任务是否完成 |
| 过程层 | 工具选择、参数、顺序、次数是否正确 |
| 安全层 | 是否越权、泄密、无限循环、误操作 |

关键观点：
```text
Agent最终答案对了，但过程错了，也不一定算通过。比如误调用了高风险工具、访问了无权限数据、重复提交表单，即使最终页面看起来成功，也有严重风险。
```

## 四、工具调用怎么校验

检查点：
- 是否调用了正确工具。
- 参数是否正确。
- 调用顺序是否合理。
- 是否遗漏必要工具。
- 是否重复调用。
- 工具失败后是否重试或降级。
- 是否调用了越权工具。

示例：

```json
{
  "task": "生成接口自动化用例",
  "expected_tools": ["read_api_doc", "search_existing_cases", "write_test_case", "run_pytest"],
  "forbidden_tools": ["delete_file", "skip_test"],
  "max_steps": 12
}
```

工具调用验证模板：

| 验证项 | 断言方式 |
|--------|----------|
| 工具选择 | expected_tools必须被调用，forbidden_tools不能被调用 |
| 参数正确 | URL、文件路径、账号、环境、请求体字段和业务ID必须符合预期 |
| 顺序正确 | 先读上下文，再生成，再执行，再修复，不能先写文件或直接提交 |
| 次数合理 | 设置max_steps、max_retries，避免无限循环和成本失控 |
| 结果可验 | 每个工具返回要记录success/error、关键输出和下一步依据 |
| 失败恢复 | 工具失败后要重试、降级或停止，不能伪造成功 |
| 权限边界 | 生产接口、删除、支付、发邮件等高风险工具必须禁止或二次确认 |

背诵表达：
```text
Agent工具调用不能只看最后结果，我会看轨迹。轨迹里要能看到它选了什么工具、参数对不对、顺序是否符合业务流程、失败后有没有合理重试，以及有没有调用越权工具。最终结果对但中间越权或重复提交，也不能算通过。
```

## 五、Agent常见风险

| 风险 | 表现 | 测试方法 |
|------|------|----------|
| 错误规划 | 步骤漏掉 | 轨迹检查 |
| 工具误用 | 参数错、工具错 | mock工具返回、断言参数 |
| 无限循环 | 反复尝试同一步 | 最大步数、循环检测 |
| 越权操作 | 调用高权限工具 | 权限矩阵 |
| 成本失控 | token/工具调用过多 | 成本上限 |
| 上下文污染 | 被网页/文档恶意指令影响 | 间接注入样本 |
| 状态污染 | 记忆残留影响下一任务 | 会话隔离 |

## 六、Agent自动化测试怎么做

```text
任务集 -> 执行Agent -> 记录轨迹 -> 校验工具调用 -> 校验最终结果 -> 统计成功率/失败原因 -> 回归
```

指标：
- 任务成功率。
- 平均步骤数。
- 工具调用准确率。
- 参数正确率。
- 重试次数。
- 失败恢复率。
- 越权调用率。
- 平均成本。
- 平均耗时。

## 七、Agent辅助测试系统架构

真实面试问过，必须背熟。

```text
输入层：需求文档、接口文档、cURL、抓包数据、已有用例、Java Controller源码
上下文层：项目规则、接口索引、历史用例、业务规范、错误日志
Agent决策层：理解任务、拆分步骤、选择工作流、生成或维护用例
工具执行层：文件读写、接口索引查询、mitmproxy抓包、pytest执行、Git diff检查
反馈闭环层：pytest失败日志回传、分析原因、修复用例、再次执行
质量护栏：前置门禁、写入位置确认、禁止跳过失败、人工确认业务断言
输出层：接口方法、pytest用例、执行报告、问题记录、提效数据
```

回答模板：
```text
我设计的Agent辅助测试系统可以分成输入、上下文、Agent决策、工具执行和反馈闭环几层。输入支持需求、接口文档、cURL、抓包和已有用例；上下文层提供项目规则、接口索引和历史用例；Agent层负责选择工作流；工具层执行文件读写、抓包分析和pytest；最后通过pytest失败日志回传形成修复闭环。最关键的是加护栏，比如必须确认写入位置、失败不能跳过、关键业务断言人工确认。
```

结合harness/loop engineering补充：

```text
我会把Agent外面那层工程化控制叫harness，把多轮执行和修复叫loop。harness负责限制目录、工具白名单、写入门禁、pytest执行和Git diff检查；loop负责根据失败日志重新规划和修复，直到通过或达到停止条件。这样Agent不是自由发挥，而是在测试工程约束里完成任务。
```

## 八、browser-use类Agent怎么讲

```text
browser-use类项目可以理解为浏览器Agent：LLM理解任务，观察页面状态，规划下一步动作，通过Playwright执行点击、输入、跳转，再读取页面反馈继续循环，直到任务完成或失败。
```

测试重点：
- 任务是否成功。
- 每一步动作是否合理。
- 元素识别是否准确。
- 页面变化后能否恢复。
- 是否有截图、录屏、日志。
- 是否避免提交/删除/支付等高风险误操作。

## 九、结合个人项目

```text
我的AI+接口自动化SKILL和AI+UI自动化/browser-use都可以按Agent思路表达。接口SKILL是文件和pytest工具型Agent，核心是用规则、上下文和执行闭环控制AI生成用例；browser-use是浏览器操作型Agent，核心是观察页面、规划动作、调用Playwright执行并记录轨迹。两者共同点是不能只看最终结果，必须看中间过程和安全边界。
```

STAR模板：

```text
S：低代码平台接口和页面复杂，手工维护自动化用例成本高，普通LLM直接生成又容易缺上下文、缺验证。
T：我要把AI能力做成可落地的测试Agent，而不是只让AI写一段代码。
A：接口侧我设计SKILL工作流，让Agent读取抓包/cURL/源码/已有用例，生成接口方法和pytest用例，再运行pytest并根据日志修复；UI侧基于browser-use二开，让Agent观察页面、规划动作并调用Playwright执行，同时记录轨迹。
R：接口用例编写效率提升93.43%，综合效率提升46.48%；browser-use项目也形成了多LLM集成和浏览器动作闭环的实践经验。
```

## 十一、回答模板

```text
Agent测试我会按结果、过程、安全三层来测。结果层看任务是否完成；过程层看工具调用轨迹，包括选了什么工具、参数是否正确、调用顺序是否合理、有没有重复或遗漏；安全层看是否越权、泄露敏感数据、陷入循环或执行高风险动作。比如browser-use这类项目，不能只看最后页面是否成功，还要记录每一步浏览器动作、截图、失败重试和日志，这样才能判断是模型规划错、元素识别错、工具执行错还是页面状态变化导致的问题。
```
