## 二、测试流程

标准表达：

```text
需求分析 -> 测试计划 -> 测试设计 -> 测试执行 -> 缺陷提交与跟踪 -> 回归测试 -> 测试报告
```

测试开发面试要补充：
- 需求分析阶段识别可测性和风险。
- 测试设计阶段选择等价类、边界值、判定表、状态迁移、场景法。
- 自动化阶段把核心用例转成可回归脚本。
- 报告阶段输出通过率、缺陷、风险和上线建议。

## 三、黑盒测试方法

### 1. 等价类划分

把输入数据按规则划分为一组组“等价”的类别，每类选一个代表值。

必须分为：
- 有效等价类：符合规则的数据。
- 无效等价类：不符合规则的数据。

示例：用户名长度 6-20 位，只允许字母和数字。

| 类型 | 等价类 | 示例 |
|------|--------|------|
| 有效 | 长度6-20，字母数字 | `test123` |
| 无效 | 长度小于6 | `abc` |
| 无效 | 长度大于20 | `a12345678901234567890` |
| 无效 | 包含特殊字符 | `test@123` |
| 无效 | 为空 | `` |

你之前错题问题：
- 只说“代表一类数据”，但没有分有效/无效。
- 把字符类型和长度规则混在一起，结构不清楚。

面试表达：
```text
等价类的关键是按业务规则划分类别，并且有效和无效都要覆盖。比如用户名规则，我会分别覆盖合法用户名、过短、过长、为空、包含非法字符等类别，每类选代表值。
```

必须背熟的结构：

```text
我做等价类不会只说“选代表值”，一定会先按规则拆成有效等价类和无效等价类。有效等价类验证系统能正确接受合法输入；无效等价类验证系统能拦截非法输入，并给出正确提示。一个字段如果同时有长度、类型、必填、唯一性等规则，我会按规则分别拆，不混在一起。
```

### 2. 边界值分析

边界值关注规则边界附近的数据。

用户名长度 6-20：

| 边界 | 用例 |
|------|------|
| 刚好小于最小值 | 5位 |
| 最小值 | 6位 |
| 最小值+1 | 7位 |
| 最大值-1 | 19位 |
| 最大值 | 20位 |
| 刚好超过最大值 | 21位 |

面试表达：
```text
边界值适合数值、长度、时间、金额、数量这类有上下限的规则，因为缺陷经常出现在边界附近。自动化里我会把边界值直接做成参数化数据。
```

等价类和边界值的区别：

```text
等价类解决“每一类规则都要覆盖”的问题，边界值解决“上下限附近最容易出错”的问题。实际设计用例时通常一起用：先按有效/无效等价类拆分类，再在有范围限制的类别里补充最小值、最大值、刚好越界这些边界数据。
```

### 3. 判定表

适合多条件组合、多业务规则。

例：优惠券是否可用。

| 条件 | 规则1 | 规则2 | 规则3 | 规则4 |
|------|------|------|------|------|
| 用户已登录 | 是 | 是 | 是 | 否 |
| 优惠券未过期 | 是 | 否 | 是 | 是 |
| 订单金额满足门槛 | 是 | 是 | 否 | 是 |
| 结果 | 可用 | 不可用 | 不可用 | 不可用 |

场景：
- 权限规则。
- 优惠券。
- 审批规则。
- 开关组合。

### 4. 状态迁移

适合订单、工单、审批、支付等状态流转。

订单状态示例：

```text
待支付 -> 已支付 -> 已发货 -> 已完成
待支付 -> 已取消
已支付 -> 退款中 -> 已退款
```

测试点：
- 合法状态能否流转。
- 非法状态是否被拦截。
- 重复操作是否幂等。
- 并发状态变更是否一致。

### 5. 场景法

适合端到端业务链路。

下单链路：
```text
登录 -> 选择商品 -> 加入购物车 -> 提交订单 -> 支付 -> 查看订单状态
```

自动化分层：
- 接口自动化覆盖业务规则和数据。
- UI自动化覆盖关键链路。
- 性能测试覆盖高频接口。

## 四、测试金字塔

```text
      UI测试：少量，核心链路
   接口/服务测试：适中，业务规则
单元测试：大量，函数/模块逻辑
```

面试表达：
```text
越底层越快、越稳定、定位越准确；越上层越接近用户，但成本高、不稳定。所以自动化策略不是把所有功能都做UI自动化，而是接口层覆盖主要业务逻辑，UI层覆盖关键用户路径。
```

## 五、需求可测性判断

一个需求可测，至少要明确：
- 输入是什么。
- 输出是什么。
- 业务规则是什么。
- 权限边界是什么。
- 异常场景是什么。
- 验收标准是什么。
- 数据状态如何变化。

面试表达：
```text
如果需求只写“支持下单”，但没说明库存不足、优惠券、支付失败、重复提交、权限等规则，我会在评审阶段提出问题，否则后续用例很难完整设计。
```

## 六、结合AI生成用例

AI可以辅助从需求生成测试点，但不能替代测试设计。

正确流程：

```text
需求文档 -> AI提取测试点 -> 人工确认业务规则 -> AI生成用例草稿 -> 按等价类/边界值/判定表补漏 -> 执行或评审 -> 失败样本回流
```

结合你的项目：
```text
我的AI+接口自动化SKILL不是让AI直接随便写用例，而是把抓包、接口文档、已有用例、Controller源码作为上下文，再要求生成后执行pytest闭环。用例设计上仍要人工确认等价类、边界值、状态流转和关键业务断言。
```

## 七、结合E10低代码平台的用例设计

你的简历重点是泛微E10低代码平台，面试官问测试理论时，要主动落到表单、流程、权限、低代码配置这些真实场景。

### 1. 表单字段规则

| 规则 | 有效等价类 | 无效等价类 | 边界值 |
|------|------------|------------|--------|
| 文本长度1-50 | 1-50位文本 | 空、51位、非法字符 | 0、1、2、49、50、51 |
| 数字范围1-999 | 1-999数字 | 0、1000、负数、非数字 | 0、1、2、998、999、1000 |
| 必填字段 | 填合法值 | 空、空格、null | 空值边界 |
| 唯一字段 | 新编码 | 重复编码 | 已存在值/新值 |

面试标准回答模板：

```text
如果让我设计E10低代码表单字段测试，我会先按字段规则拆等价类。比如文本字段长度1到50，允许普通字符且必填，我会设计有效等价类：合法长度且合法字符；无效等价类：为空、超长、非法字符、只有空格。然后补边界值：0、1、2、49、50、51。落到自动化时，这些数据会做成pytest参数化，断言不只看接口成功失败，还要检查字段配置是否保存、页面展示是否生效、数据库或配置表状态是否正确。
```

### 2. 流程与审批状态

E10流程类场景不要只测“能提交”，要测状态流转：

```text
草稿 -> 提交 -> 审批中 -> 通过 -> 归档
草稿 -> 删除
审批中 -> 驳回 -> 重新提交
审批中 -> 撤回
```

测试点：
- 合法流转是否成功。
- 非法流转是否拦截，例如已归档流程不能再次撤回。
- 重复提交、重复审批是否幂等。
- 不同节点审批人权限是否正确。
- 流程状态、待办、日志、消息通知是否一致。

### 3. 权限与角色组合

权限测试适合用判定表，不要凭感觉列几条。

| 角色 | 有菜单权限 | 有数据权限 | 有操作权限 | 预期 |
|------|------------|------------|------------|------|
| 管理员 | 是 | 是 | 是 | 可查看、可编辑 |
| 普通用户 | 是 | 部分 | 否 | 可查看部分数据、不可编辑 |
| 无权限用户 | 否 | 否 | 否 | 菜单不可见或访问被拒绝 |
| 跨部门用户 | 是 | 否 | 否 | 可进菜单但看不到非授权数据 |

面试表达：

```text
E10低代码平台权限测试我会拆成菜单权限、数据权限、操作权限三层。菜单权限看入口是否可见，数据权限看能看到哪些记录，操作权限看新增、编辑、删除、导出等按钮是否可用。用例设计上用判定表覆盖角色和权限组合，自动化断言同时检查接口返回、页面按钮、数据范围和越权访问。
```

### 4. 低代码配置生效

低代码平台的风险不只是“保存配置成功”，而是“配置是否真正影响运行态”。

| 配置类型 | 设计重点 |
|----------|----------|
| 字段配置 | 必填、默认值、长度、类型、显示隐藏 |
| 页面配置 | 布局、按钮、列表列、搜索条件 |
| 流程配置 | 节点、审批人、条件分支、驳回规则 |
| 权限配置 | 菜单、数据范围、操作按钮 |
| 报表配置 | 字段聚合、筛选条件、权限隔离 |

标准回答模板：

```text
低代码平台测试的重点是配置态和运行态一致。比如我配置了一个字段必填，不能只断言保存接口返回成功，还要进入运行页面提交空值，确认前端和后端都能拦截；配置了权限，不能只看角色保存成功，还要用不同账号验证菜单、数据和操作按钮。我的设计思路是：配置保存成功、运行态生效、异常配置被拦截、权限隔离正确、变更后回归不影响已有配置。
```

## 九、回答模板

```text
我设计用例一般先看业务风险，再选方法。字段输入类规则用等价类和边界值，多条件业务规则用判定表，订单、审批、支付这类状态变化用状态迁移，完整业务链路用场景法。落到自动化里，我会把这些测试点转成pytest参数化数据和fixture前置数据，并补充接口响应、数据库、日志或状态流转断言。AI可以帮我提高用例生成效率，但最终一定要经过人工确认业务预期和执行验证。
```

E10项目标准回答模板：

```text
结合我做过的E10低代码平台，我设计用例会先识别对象是字段、表单、页面、流程、权限还是报表。字段规则用等价类和边界值，必须区分有效等价类和无效等价类；权限组合用判定表，拆成菜单权限、数据权限和操作权限；流程审批用状态迁移，覆盖提交、通过、驳回、撤回、归档以及非法流转；完整业务链路再用场景法串起来。落到自动化时，我会用pytest参数化承载测试数据，用fixture准备账号、表单和流程配置，断言接口响应、数据库状态、页面展示和权限隔离。AI可以帮我生成草稿，但我会按这些测试设计方法补漏，并通过最小范围pytest执行验证。
```
