## 二、性能测试类型

| 类型 | 目的 | 面试表达 |
|------|------|----------|
| 基准测试 | 获取单接口/单场景基线 | 看系统正常水平 |
| 负载测试 | 目标负载下是否满足指标 | 验证预期并发 |
| 压力测试 | 找系统极限和崩溃点 | 持续加压 |
| 稳定性测试 | 长时间运行是否稳定 | 看内存泄漏、错误积累 |
| 容量测试 | 支撑多大数据量/用户量 | 规划容量 |
| 峰值测试 | 短时间峰值冲击 | 促销/高峰场景 |

你之前混淆过负载和压力：
```text
负载测试是看系统在预期负载下是否达标；压力测试是超过预期逐步加压，找到瓶颈和极限。
```

稳定性测试也要单独记住：

```text
稳定性测试不是继续加压，而是在一个相对稳定的业务负载下长时间运行，观察错误率、响应时间、资源占用是否持续恶化，比如内存泄漏、连接未释放、日志错误累积。
```

结合E10低代码平台可以这样说：

```text
如果测E10低代码平台，我会把负载测试理解为验证登录、表单保存、列表查询、报表查询、审批提交这些核心场景在预期并发下是否达标；压力测试是继续提高并发或请求频率，找到服务、数据库或网关的瓶颈；稳定性测试则是在接近业务高峰但可接受的负载下持续运行，重点看错误率、响应时间和资源是否出现劣化。这里我不会随便报线上压测数据，只讲测试方法和分析思路。
```

## 三、核心指标

| 指标 | 含义 | 注意点 |
|------|------|--------|
| TPS | 每秒事务数 | 业务事务，不一定等于请求数 |
| QPS | 每秒请求数 | 请求维度 |
| 响应时间 | 请求从发出到收到响应 | 看平均值也要看P90/P95/P99 |
| 并发用户 | 同时在线或同时请求的用户 | 不等于TPS |
| 吞吐量 | 单位时间处理的数据量 | 网络/服务能力 |
| 错误率 | 失败请求占比 | 性能达标必须错误率低 |
| CPU/内存/IO/网络 | 资源指标 | 定位瓶颈 |

面试表达：
```text
TPS受响应时间、并发用户、思考时间和服务端处理能力影响。并发用户多不代表TPS一定高，如果响应时间变长或服务端瓶颈，TPS反而可能下降。
```

关系理解：

```text
在没有思考时间、请求持续打满的简化场景下，可以粗略理解为：并发数 ≈ TPS × 平均响应时间。响应时间越长，同样TPS需要占用更多并发；如果并发继续增加但服务端已经到瓶颈，响应时间会上升，错误率可能增加，TPS可能不升反降。
```

面试追问时不要只背公式，要补一句：

```text
真实业务里还要看用户思考时间、业务链路长短、接口是否串行、是否有缓存和限流，所以我会结合压测模型和服务端监控一起判断。
```

## 四、性能测试流程

```text
明确目标 -> 设计场景 -> 准备数据 -> 编写脚本 -> 单用户调试 -> 梯度加压 -> 监控资源 -> 分析瓶颈 -> 输出报告
```

每一步要说什么：
- 目标：TPS、响应时间、错误率、资源阈值。
- 场景：登录、查询、保存、审批、报表等业务比例。
- 数据：避免缓存命中假象和数据重复失败。
- 监控：应用、数据库、中间件、服务器。
- 报告：结论、瓶颈、证据、建议。

## 五、JMeter必背

### 1. Test Plan组件

| 组件 | 作用 |
|------|------|
| Thread Group | 线程数、Ramp-up、循环次数 |
| Sampler | 发请求，如HTTP Request |
| Controller | 控制执行逻辑 |
| Timer | 模拟思考时间 |
| Assertion | 响应断言 |
| Config Element | 公共配置，如HTTP Header、CSV Data Set |
| Listener | 查看和保存结果 |

### 2. 线程组参数

| 参数 | 含义 |
|------|------|
| Number of Threads | 虚拟用户数 |
| Ramp-up | 在多长时间内启动全部线程 |
| Loop Count | 每个线程循环次数 |
| Duration | 持续运行时间 |

面试表达：
```text
如果线程数100，Ramp-up 50秒，表示50秒内逐步启动100个虚拟用户，平均每秒启动2个，避免瞬间冲击。
```

线程组追问模板：

```text
线程数决定同时模拟多少虚拟用户，Ramp-up决定这些用户启动得快还是慢，Loop Count决定每个用户循环多少次，Duration适合控制压测持续时间。实际压测时我会先低并发调通脚本，再按梯度增加线程数，观察TPS、响应时间、错误率和服务端资源变化。
```

### 3. 聚合报告指标

| 指标 | 含义 | 面试注意点 |
|------|------|------------|
| Samples | 请求样本数 | 样本太少结论不可靠 |
| Average | 平均响应时间 | 不能只看平均值 |
| Median | 中位数响应时间 | 50%请求低于该值 |
| 90%/95%/99% Line | 分位响应时间 | 更能反映大部分用户体验 |
| Min/Max | 最小/最大响应时间 | Max异常要结合日志分析 |
| Error % | 错误率 | 错误率高时TPS没有意义 |
| Throughput | 吞吐量 | 可近似看成请求吞吐能力，不一定等于业务TPS |
| Received/Sent KB/sec | 网络收发速率 | 用于判断网络或响应体大小影响 |

聚合报告回答模板：

```text
JMeter聚合报告我不会只看Average，我会重点看Throughput、Error%、P90/P95/P99响应时间。平均响应时间可能被少量快请求拉低，P95更接近大多数用户体验。如果TPS下降同时P95变高、错误率上升，我会继续结合服务端资源、慢SQL、连接池和日志定位。
```

### 4. 正式压测为什么不用GUI

```text
JMeter GUI适合脚本调试，正式压测应使用Non-GUI模式，减少GUI监听器带来的资源消耗和结果干扰。
```

命令：

```bash
jmeter -n -t test.jmx -l result.jtl -e -o report
```

## 六、Locust必背

Locust用Python代码表达用户行为，适合Python测试工程师和复杂业务流。

示例：

```python
from locust import HttpUser, task, between

class WebsiteUser(HttpUser):
    wait_time = between(1, 3)

    def on_start(self):
        self.client.post("/login", json={"username": "test", "password": "123456"})

    @task(3)
    def list_orders(self):
        self.client.get("/api/orders", name="/api/orders")

    @task(1)
    def create_order(self):
        self.client.post("/api/orders", json={"skuId": "sku001", "count": 1})
```

结合E10低代码平台的表达示例（路径只表示脚本写法，面试时替换成真实接口名）：

```python
from locust import HttpUser, task, between

class EBuilderUser(HttpUser):
    wait_time = between(1, 3)

    def on_start(self):
        # 登录后保存token或cookie，后续请求复用会话
        self.client.post("/login", data={"username": "test", "password": "******"}, name="登录")

    @task(5)
    def query_form_list(self):
        self.client.get("/api/form/list", name="表单列表查询")

    @task(3)
    def save_form_data(self):
        payload = {"formId": "demo_form", "fields": {"name": "test"}}
        self.client.post("/api/form/save", json=payload, name="表单数据保存")

    @task(1)
    def query_report(self):
        self.client.get("/api/report/query", name="报表查询")
```

这段脚本面试要讲清楚：

```text
HttpUser表示一个虚拟用户，on_start做登录和会话准备，@task权重表达不同业务操作的比例，wait_time模拟用户思考时间。比如低代码平台里，列表查询可能比保存和报表查询更高频，所以权重可以设置得更高。真实压测前还要准备参数化数据，避免所有用户使用同一条数据导致缓存或数据冲突。
```

必背概念：
- `HttpUser`：模拟HTTP用户。
- `@task`：定义用户行为，可配置权重。
- `wait_time`：模拟用户思考时间。
- `on_start`：用户启动时执行，如登录。
- 分布式压测：多worker扩展压力。

你当前Locust薄弱，面试可以先讲清楚核心结构，不要强装复杂脚本。

## 七、瓶颈定位

TPS下降常见原因：

| 层级 | 可能原因 | 怎么查 |
|------|----------|--------|
| 应用 | CPU高、线程池满、GC、锁竞争 | 监控、日志、线程 |
| 数据库 | 慢SQL、索引缺失、锁等待、连接池满 | 慢查询、EXPLAIN、连接数 |
| 缓存 | 命中率低、内存满、热点key | Redis指标 |
| MQ | 队列积压、消费慢 | 队列深度、消费日志 |
| 网关 | 连接数、限流、502/504 | Nginx日志 |
| 第三方 | 依赖服务慢 | 调用链、超时日志 |
| 网络 | 带宽、延迟 | 网络监控 |

### 性能瓶颈定位五步法（必背）

```text
第一步看现象：TPS是否下降、响应时间是否升高、错误率是否增加，先判断是慢、错，还是吞吐上不去。
第二步分层定位：从压测端、网关、应用、数据库、中间件、服务器资源逐层排除。
第三步找证据：看CPU、内存、IO、网络、线程池、连接池、GC、慢SQL、错误日志和调用链。
第四步做验证：调整一个变量复测，比如增加连接池、优化SQL、加索引、调整线程数或隔离慢接口，确认瓶颈是否变化。
第五步出结论：说明瓶颈位置、证据、影响范围、优化建议和复测结果。
```

TPS下降定位模板：

```text
如果TPS下降，我会先看错误率和响应时间。如果错误率升高，优先查超时、连接拒绝、限流、502/504等错误；如果错误率不高但响应时间升高，就重点查应用线程池、数据库慢SQL、连接池、锁等待、GC、IO和外部依赖。如果CPU不高但响应慢，通常不能只说机器没压力，还要排查数据库、网络、锁等待、连接池耗尽、下游服务慢或线程阻塞。
```

面试表达：
```text
性能瓶颈不能只看压测工具报告。JMeter/Locust告诉我现象，比如响应时间变长、TPS下降、错误率升高；真正定位要结合应用、数据库、中间件、服务器和日志。
```

## 八、性能报告怎么写

报告结构：
1. 测试目标。
2. 测试环境。
3. 测试数据量。
4. 场景模型和并发模型。
5. 压测脚本说明。
6. 指标结果：TPS、响应时间、错误率、资源。
7. 瓶颈分析。
8. 结论：是否达标。
9. 优化建议。
10. 风险和遗留问题。

## 十一、回答模板

### 1. 通用回答模板

```text
性能测试我会先明确目标，比如目标TPS、P95响应时间、错误率和资源阈值，然后按真实业务比例设计场景，准备数据，编写JMeter或Locust脚本，先单用户调试，再逐步加压。分析时不会只看工具报告，还会结合CPU、内存、IO、数据库慢SQL、连接池、缓存、MQ、网关日志等定位瓶颈。JMeter偏配置化，适合快速压HTTP接口；Locust偏Python代码化，适合复杂业务流。最终报告要给出环境、场景、指标、瓶颈、结论和优化建议。
```

### 2. 负载/压力/稳定性模板

```text
负载测试是验证系统在预期业务负载下能不能满足指标；压力测试是超过预期继续加压，找到系统瓶颈和极限；稳定性测试是在相对稳定的负载下长时间运行，看响应时间、错误率、资源占用是否持续恶化。简单说，负载看达标，压力找极限，稳定性看长期是否可靠。
```

### 3. JMeter模板

```text
JMeter里我重点关注线程组和聚合报告。线程组里线程数代表虚拟用户数，Ramp-up控制启动速度，Loop Count或Duration控制执行次数和持续时间。聚合报告里我不会只看平均响应时间，会重点看Throughput、Error%、P90/P95/P99。正式压测一般用Non-GUI模式，避免GUI监听器消耗资源影响结果。
```

### 4. Locust模板

```text
Locust适合用Python代码表达用户行为。HttpUser表示虚拟用户，on_start一般做登录和会话初始化，@task定义业务动作并用权重表示业务比例，wait_time模拟用户思考时间。结合我的接口自动化经验，Locust脚本可以复用接口请求、鉴权、参数化数据和业务链路，所以对Python测试工程师比较友好。
```

### 5. TPS下降瓶颈定位模板

```text
如果压测中TPS下降，我会先看错误率、响应时间和压测端是否正常，确认不是脚本或压测机瓶颈。然后分层排查网关、应用、数据库、中间件和服务器资源。重点看CPU、内存、IO、网络、线程池、连接池、GC、慢SQL、锁等待、缓存命中率和错误日志。定位后只改一个关键变量复测，最后给出证据、结论和优化建议。
```

### 6. 项目经验模板

```text
结合我在泛微E10低代码平台的经验，我会优先选择登录、表单保存、列表查询、报表查询、审批提交这类核心链路做性能场景。我的优势是接口自动化和Python能力比较强，可以把已有接口请求、测试数据、鉴权逻辑沉淀到JMeter或Locust脚本里，再通过测试平台、日志和服务端监控辅助分析。具体线上TPS数据我不会虚报，但我能讲清楚场景设计、脚本表达、指标分析和瓶颈定位流程。
```
