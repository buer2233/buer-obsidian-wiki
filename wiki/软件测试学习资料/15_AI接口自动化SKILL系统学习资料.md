```text
AI+接口自动化SKILL系统，是我基于Claude Code/Codex和现有pytest接口自动化框架设计的一套AI辅助接口用例编写与维护流程，用规则、上下文、工具和pytest执行闭环，让AI稳定生成可执行的接口方法和自动化用例。
```

面试加强版：

```text
这个项目不是单纯让AI帮我写几段代码，而是把Claude Code、Codex和SKILLS规则体系接入到E10低代码平台的接口自动化流程里。AI负责根据抓包、cURL、已有用例或Java Controller源码生成候选接口方法和pytest用例，我负责把输入、输出、执行、失败修复和验收规则固化成闭环，保证最后进入用例库的是执行通过、断言可信、diff可审查的测试资产。
```

## 三、项目背景

业务背景：
- 泛微E10低代码平台接口数量多。
- 表单、页面、报表、应用、菜单、数据仓库等模块回归压力大。
- 手工写接口方法和pytest用例耗时。
- 维护失败用例需要反复看日志、响应和历史用例。

痛点：
- 接口用例产出慢。
- 用例结构重复，适合AI辅助。
- 业务断言不能完全交给AI，需要执行验证和人工确认。
- 直接vibe coding容易乱改文件、漏跑测试、跳过失败。

## 五、4种新增用例工作流

| 工作流 | 输入 | 适用场景 |
|--------|------|----------|
| 抓包驱动 | mitmproxy真实UI流量 | 复杂链路、新接口多 |
| 参考已有用例 | 相似接口方法和用例 | 结构相似接口 |
| cURL手工 | cURL请求 | 单接口快速补用例 |
| Java Controller源码参考 | Controller源码 | 查未覆盖接口、理解入参 |

表达重点：
```text
抓包驱动适合从真实页面操作还原接口链路；参考已有用例能保证风格一致；cURL适合单接口快速补充；Controller源码适合发现未覆盖接口，但真实payload和断言仍要以抓包、响应和已有用例为准。
```

结合真实项目说法：

| 工作流 | 我在E10项目里的用法 | 质量门禁 |
|--------|--------------------|----------|
| 抓包驱动 | 在e-Builder页面完成表单、页面、报表等真实操作，用mitmproxy抓取接口链路，再让AI提取关键请求和依赖关系 | 必须保留真实请求顺序、登录态、前置数据和关键响应断言 |
| 参考已有 | 找同模块历史接口方法和pytest用例，让AI按项目已有命名、目录和断言风格补新用例 | 生成后对比已有风格，避免AI自造结构 |
| cURL手工 | 单接口临时验证或快速补覆盖时，把cURL作为最小输入交给AI生成方法和用例 | 需要补充业务断言，不能只断言status_code |
| Java Controller源码 | 根据Controller方法识别接口路径、参数、权限和未覆盖接口 | payload和预期结果不能只信源码，要结合抓包和页面复测 |

一句话记忆：

```text
抓包保证真实，参考已有保证一致，cURL保证快速，Controller保证覆盖；最后都要回到pytest执行和失败修复闭环。
```

## 七、质量保障机制

| 风险 | 你的控制方式 |
|------|--------------|
| AI不知道写哪 | 前置门禁，明确接口方法和用例位置 |
| AI上下文过长 | SKILL拆分，按需读取 |
| AI断言太浅 | 参考已有用例，人工确认业务断言 |
| AI不跑测试 | 强制最小范围pytest |
| AI跳过失败 | 规则禁止skip/xfail掩盖失败 |
| AI乱改文件 | Git diff检查 |
| BUG和用例维护边界不清 | 页面复测、需求确认、人工判断 |

面试表达：
```text
我把AI生成用例看成候选产物，不是最终结果。必须经过pytest执行、失败修复、Git diff检查和人工确认关键业务断言，才能进入自动化用例库。
```

AI生成用例后的执行/失败修复闭环：

```text
生成接口方法和用例
  -> 执行最小范围pytest
  -> 收集失败日志、请求参数、响应体、堆栈
  -> 判断失败类型：功能BUG / 用例问题 / 环境问题 / 数据问题
  -> 只在明确是用例问题时修改用例
  -> 重新执行pytest
  -> Git diff审查
  -> 人工确认关键断言后合入
```

面试追问“AI生成的东西怎么保证可靠”时，不要只回答“我会人工检查”，要按三层回答：

```text
第一层是输入约束：SKILL要求AI读取正确的抓包、已有用例、接口方法和项目规则。
第二层是执行约束：生成后必须跑最小范围pytest，失败不能skip或xfail绕过。
第三层是业务约束：关键断言、BUG判断和用例维护边界由人工结合页面复测、需求和历史行为确认。
```

## 十、遇到的难点案例

### 1. AI上下文过长导致漏步骤

```text
现象：AI生成用例后容易漏掉pytest执行或某些检查项。
根因：SKILL内容过长，低频规则占用上下文。
解决：拆分SKILL子文档，按工作流渐进式读取。
结果：AI执行更聚焦，遗漏减少。
```

STAR可背模板：

```text
S：E10低代码平台接口多、模块多，我想用Claude Code/Codex提升接口用例产出。
T：我的任务不是让AI一次性写代码，而是把AI稳定接入公司已有pytest接口自动化框架。
A：我把规则拆成SKILLS和工作流文档，按抓包、参考已有、cURL、Controller源码四种场景渐进式加载；同时增加写入位置确认、pytest执行、失败修复、Git diff检查这些门禁。
R：最终接口用例编写效率从5周198条提升到383条，提升约93%，接口方法+用例综合产出从327提升到479，提升约46%。
```

### 2. 模型为了通过而跳过失败

```text
现象：pytest失败后，模型尝试skip失败用例。
根因：规则没有明确禁止用跳过掩盖问题。
解决：增加规则，失败必须分析原因，不能skip/xfail绕过。
```

### 3. BUG和用例维护边界不清

```text
现象：接口返回变化导致用例失败，AI倾向直接改断言。
解决：要求AI给出页面复测步骤和判断依据，再结合需求确认。
结论：数据为0应返回0不是空，最终作为BUG提交。
```
