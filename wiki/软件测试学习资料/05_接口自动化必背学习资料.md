## 二、接口测试测什么

接口测试不是只测“能不能请求成功”，至少覆盖：

| 维度 | 说明 |
|------|------|
| 请求方法 | GET/POST/PUT/PATCH/DELETE是否符合设计 |
| 参数 | 必填、类型、边界、非法值、组合 |
| 鉴权 | token、cookie、session、权限、越权 |
| 响应 | 状态码、业务码、字段、类型、结构 |
| 业务规则 | 状态流转、金额、库存、优惠、审批 |
| 数据库 | 落库、更新、删除、状态一致 |
| 幂等 | 重复请求结果是否符合设计 |
| 并发 | 是否超卖、重复创建、数据不一致 |
| 安全 | 越权、敏感字段、限流、重放 |
| 日志 | 错误日志、trace id、调用链 |

## 三、HTTP方法和状态码

### 1. 请求方法

| 方法 | 语义 | 面试表达 |
|------|------|----------|
| GET | 查询资源 | 参数通常在query，语义只读 |
| POST | 创建、提交、动作 | 默认不幂等，如下单、登录 |
| PUT | 整体替换或创建 | 幂等，重复执行结果一致 |
| PATCH | 局部更新 | 修改部分字段 |
| DELETE | 删除资源 | 幂等语义 |

你之前混淆过 POST/PUT/PATCH，必须背熟：
```text
POST偏提交和创建，不保证幂等；PUT偏整体替换，通常幂等；PATCH偏局部更新。
```

### 2. 状态码

| 状态码 | 含义 | 测试定位 |
|--------|------|----------|
| 200 | 成功 | 常规成功 |
| 201 | 已创建 | 创建资源成功 |
| 204 | 成功无响应体 | 删除或无内容返回 |
| 400 | 请求语法/参数错误 | 参数格式、必填 |
| 401 | 未认证 | 未登录、token无效 |
| 403 | 已认证但无权限 | 权限不足 |
| 404 | 资源不存在 | URL或资源ID错误 |
| 405 | 方法不允许 | GET/POST用错 |
| 409 | 资源冲突 | 重复创建、状态冲突 |
| 422 | 语义无法处理 | 参数合法但业务校验失败 |
| 429 | 请求过多 | 限流 |
| 500 | 服务端异常 | 看服务日志 |
| 502 | 网关收到无效上游响应 | 网关/上游服务 |
| 503 | 服务不可用 | 服务过载/维护 |
| 504 | 网关等待上游超时 | 上游慢或不可达 |

### 3. HTTP状态码和业务错误码

```text
HTTP状态码表示协议层和请求处理的大类结果，业务错误码表示具体业务规则结果。比如HTTP 200但业务码为库存不足，是协议请求成功但业务失败。
```

## 四、服务端下单接口怎么测

真实面试已问过，必须作为P0背熟。

### 1. 核心对象

- 用户
- 商品/SKU
- 库存
- 价格
- 优惠券
- 地址
- 支付方式
- 订单
- 订单明细
- 库存流水
- 支付流水

### 2. 正常场景

| 场景 | 断言 |
|------|------|
| 单商品下单 | 返回订单号，订单表正确 |
| 多商品下单 | 明细数量正确 |
| 使用优惠券 | 优惠金额正确 |
| 不使用优惠券 | 原价正确 |
| 不同支付方式 | 支付状态和订单状态正确 |

### 3. 参数异常

- 商品ID为空/不存在。
- SKU ID不存在。
- 数量为0、负数、超大。
- 地址为空或无效。
- 优惠券为空、过期、不属于当前用户。
- 支付方式不支持。
- 篡改价格、用户ID、订单金额。

### 4. 业务异常

- 库存不足。
- 商品下架。
- 价格变更。
- 重复提交。
- 订单超时。
- 支付失败。
- 优惠券已使用。

### 5. 幂等与并发

重点表达：
```text
下单接口通常是POST，默认不幂等，但业务上需要通过幂等键、订单号、请求流水或防重复提交机制保证重复请求不会生成多笔错误订单，也不能重复扣库存。
```

并发测试：
- 最后一件库存，多个用户同时下单。
- 同一用户连续点击提交。
- 支付回调重复通知。

断言：
- 库存不能扣成负数。
- 有效订单数量符合设计。
- 重复请求结果一致或返回重复提交提示。

### 6. 数据校验

不要只看响应：
- 订单表。
- 订单明细表。
- 库存表。
- 优惠券状态。
- 支付流水。
- 日志/trace id。
- 异步消息。

## 五、接口自动化落地

推荐结构：

```text
fixture准备用户/商品/库存 -> 调用下单接口 -> 断言响应 -> 查询数据库 -> 校验库存/订单/优惠券 -> 清理数据
```

pytest示例：

```python
@pytest.mark.parametrize("count, expected_code", [
    (1, "SUCCESS"),
    (0, "INVALID_COUNT"),
    (-1, "INVALID_COUNT"),
])
def test_create_order(order_api, db, sku, count, expected_code):
    resp = order_api.create_order(sku_id=sku.id, count=count)
    body = resp.json()
    assert body["code"] == expected_code
    if expected_code == "SUCCESS":
        order = db.get_order(body["data"]["orderId"])
        assert order["sku_id"] == sku.id
```

### 1. 结合我的E10真实项目怎么讲

可背表达：
```text
我做接口自动化主要是在泛微E10数字化运营平台的低代码模块里落地，核心对象包括表单、页面、报表、应用、菜单、数据仓库等。我的思路不是单纯把接口调通，而是把真实业务链路拆成可维护的pytest用例：先准备测试账号、组织权限和基础数据，再用requests发起接口请求，最后校验响应、业务状态和必要的数据落库。对于低代码平台这种配置型系统，我会特别关注不同角色权限、不同表单配置、不同环境数据差异，以及接口之间的状态流转。
```

这段回答适合回答：
- 你做过什么接口自动化项目？
- 你的接口自动化不是只写Demo，怎么证明有真实业务？

### 2. pytest + requests 分层模板

面试讲结构，不需要现场写完整代码：

```text
我的接口自动化一般分为四层：第一层是api封装，把requests调用封装成业务方法；第二层是case层，用pytest组织测试用例；第三层是data层，管理账号、环境、参数化数据；第四层是common层，放鉴权、数据库、日志、断言和报告工具。这样接口地址变更、token逻辑变更、断言规则变更时，不需要大面积改用例。
```

可背关键词：
- `requests.Session()`：复用连接、统一headers/cookies/token。
- `pytest fixture`：准备登录态、测试数据、环境变量、清理动作。
- `pytest.mark.parametrize`：做数据驱动。
- `conftest.py`：放公共fixture和hook。
- `pytest-html/allure`：生成报告，便于Jenkins归档。

### 3. 数据驱动怎么回答

```text
数据驱动不是把所有参数塞进Excel就完了，我会先按业务规则设计数据分类。比如下单接口会分正常数据、必填缺失、类型错误、边界值、权限数据、状态冲突数据、幂等数据和安全数据。实现上可以用pytest parametrize、YAML/JSON/Excel或数据库读取，具体选型看团队维护成本。我的倾向是稳定核心用例直接写在代码或结构化文件里，变化频繁的业务数据从平台或数据库读取。
```

常见追问：
```text
如果测试数据互相影响，我会通过fixture做前置数据准备和后置清理；如果不能删除真实业务数据，就使用独立测试账号、独立命名规则、状态回滚或环境隔离。
```

### 4. 测试账号和环境管理

```text
接口自动化里账号和环境管理很关键。我一般会把环境地址、数据库、账号、token、租户或组织信息独立配置，不硬编码在用例里。测试账号会按角色区分，比如管理员、普通用户、无权限用户、跨组织用户，用来覆盖鉴权和越权场景。多环境执行时，通过命令行参数或配置文件切换dev/test/pre环境，避免同一套用例在不同环境里改代码。
```

可背要点：
- 账号分层：管理员、普通用户、只读用户、无权限用户、跨部门/跨租户用户。
- 环境分层：base_url、数据库连接、账号密码、文件服务、Mock服务。
- 安全处理：密码、token、数据库密码不提交到Git，使用环境变量或CI凭据。
- 数据隔离：测试数据加固定前缀，执行后清理或定期清理。

## 六、API安全测试

结合OWASP API Security：

| 风险 | 测试点 |
|------|--------|
| BOLA对象级越权 | A用户访问B用户订单 |
| Broken Authentication | token过期、伪造、缺失 |
| Object Property Authorization | 篡改价格、角色、用户ID |
| Resource Consumption | 限流、超大分页、批量请求 |
| Function Level Authorization | 普通用户调用管理员接口 |

面试表达：
```text
接口测试除了功能，还要测认证、授权、越权、敏感字段、限流和幂等。比如订单接口一定要测跨用户订单ID、伪造userId、篡改金额等安全风险。
```

## 七、结合AI+接口自动化SKILL

你的项目表达：

```text
对于复杂业务链路，我会先通过mitmproxy抓取真实页面操作产生的接口，再筛选关键接口，让AI结合已有用例和项目规则生成pytest用例。生成后必须执行最小范围pytest，根据真实响应补充断言，最后人工确认订单、库存、金额这类关键业务断言。
```

这能回答：
- AI怎么提升接口自动化效率？
- AI生成用例如何保证质量？
- 抓包驱动适合什么场景？
- 为什么不是只用AI写代码？

### 1. 四种AI工作流必须背熟

| 工作流 | 适用场景 | 面试回答 |
|--------|----------|----------|
| 抓包驱动 | 页面操作链路复杂、接口文档不完整 | 先用mitmproxy抓真实请求，再让AI识别关键接口、参数依赖和断言点 |
| 参考已有用例 | 项目已有同类接口用例 | 让AI按现有目录、fixture、断言风格生成，保证风格一致 |
| cURL手工 | 单个接口临时补充或接口来自调试工具 | 把cURL转换成requests/pytest用例，再补充参数化和断言 |
| Java Controller源码 | 后端接口逻辑清楚但文档缺失 | 参考Controller入参、权限注解、业务分支生成测试点和用例草稿 |

### 2. AI生成用例如何保证质量

可背表达：
```text
我不会直接信任AI生成的用例，而是做闭环校验。第一步限制AI读取范围，只给它接口规范、已有用例、fixture和必要源码；第二步要求它按项目模板生成pytest用例；第三步本地执行最小范围pytest，确认路径、fixture、账号和环境都能跑通；第四步人工重点检查业务断言，比如权限、状态流转、金额、数据落库；最后把稳定用例合入框架。这样AI负责提效，人负责质量兜底。
```

### 3. 和成都10-20K岗位匹配的表达

```text
我这块优势不是只会requests调接口，而是把AI融入到了真实接口自动化流程里。传统方式下，一个复杂接口用例需要人工看页面、抓包、分析参数、写代码和调试；我做的SKILL把这些步骤沉淀成标准工作流，让AI按项目规则生成用例草稿，再通过pytest执行结果反向修正。这个方案在公司内部推广后，接口用例编写效率提升过93%，综合效率提升46%。面试中如果需要，我可以重点讲抓包驱动和参考已有用例这两种最常用流程。
```

## 九、回答模板

### 1. 通用接口自动化模板

```text
接口自动化我会从请求、鉴权、业务规则、断言、数据一致性和安全几个层面测试。以下单接口为例，不只是看返回200，而是要验证商品、库存、金额、优惠、订单状态、数据库落库和幂等。正常场景测下单成功，异常场景测库存不足、商品下架、价格变更、优惠券不可用、参数缺失，安全场景测未登录、越权、篡改金额和用户ID。自动化上我用pytest + requests，通过fixture准备数据，参数化覆盖场景，断言响应、数据库和状态流转。复杂链路可以结合我做过的AI+接口自动化SKILL，通过抓包生成用例草稿，再执行pytest闭环验证。
```

### 2. 结合E10项目模板

```text
我在E10低代码平台里主要做表单、页面、报表、应用等模块的接口自动化。低代码平台接口的特点是配置多、权限多、数据状态多，所以我会先把业务对象和角色账号梳理清楚，再用pytest fixture准备前置数据，用requests或封装后的api层调用接口，最后校验响应、数据库和业务状态。对于新增接口，我会优先参考已有同类用例，保持fixture、断言和报告风格一致；对于复杂链路，会通过抓包还原真实用户操作，再沉淀成稳定的自动化用例。
```

### 3. 数据驱动模板

```text
我理解的数据驱动是把测试逻辑和测试数据分离，但前提是数据分类要设计清楚。比如一个接口我会按正常值、必填缺失、类型错误、边界值、权限数据、状态冲突和安全数据来组织，再通过pytest parametrize或外部YAML/JSON加载。这样新增场景时主要补数据，不需要复制大量测试代码。对于账号、token和环境变量，我会通过fixture和配置文件统一管理，避免写死在用例里。
```

### 4. AI+接口自动化模板

```text
AI+接口自动化这块是我现在比较有差异化的经验。我做过基于Claude Code的接口用例编写SKILL，支持抓包驱动、参考已有用例、cURL手工和Java Controller源码四种工作流。我的定位是让AI完成重复性的分析和代码草稿生成，但质量控制仍然依赖pytest执行、真实响应校验和人工业务断言检查。这个方式在真实项目里提升过接口用例编写效率，比较适合接口多、规则相似、历史用例较多的团队。
```

### 5. 面试官追问“你具体负责什么”

```text
我主要负责把接口自动化从零散脚本推进到框架化和AI辅助生成。具体包括维护pytest + requests框架、封装公共fixture、管理测试账号和环境、设计接口断言规则、把执行结果接入报告，同时把AI工作流沉淀成SKILL在团队内部推广。涉及核心业务断言时，我会自己确认规则，不会完全交给AI。
```
