# 性能测试学习文件

## 1. 性能测试面试题

### 基础概念

**Q: 什么是性能测试？**

A: 性能测试是通过自动化测试工具模拟多种正常、峰值及异常负载条件来对系统的各项性能指标进行测试。

**Q: 性能测试包含哪些类型？**

A: 主要包括：
- 负载测试：验证系统在预期负载下的表现
- 压力测试：验证系统在极限负载下的表现
- 稳定性测试：验证系统长时间运行的稳定性
- 并发测试：验证系统并发处理能力

**Q: 性能测试关注哪些指标？**

A: 核心指标：
- 响应时间：从请求发出到收到响应的时间
- 吞吐量：单位时间处理的请求数
- 并发用户数：同时在线的用户数
- TPS：每秒事务数
- 错误率：失败请求占比
- 资源利用率：CPU、内存、磁盘、网络使用率

**Q: 如何确定性能测试的并发用户数？**

A: 常用方法：
- 经验公式：并发用户数 = 在线用户数 * (5%~20%)
- 80/20原则：80%的请求集中在20%的时间内
- 峰值估算：根据业务峰值日志分析

### 工具相关

**Q: JMeter的工作原理是什么？**

A: JMeter通过创建多个线程来模拟用户请求，每个线程独立执行测试计划中定义的请求，收集并统计响应数据。

**Q: JMeter中如何实现参数化？**

A: 常用方式：
- CSV Data Set Config：从CSV文件读取
- 用户定义的变量：全局变量
- 函数助手：随机数、时间等
- 数据库查询：从数据库获取

**Q: JMeter中如何做关联？**

A: 使用后置处理器：
- 正则表达式提取器
- JSON提取器
- XPath提取器
- 边界值提取器

**Q: 如何分析性能测试结果？**

A: 分析步骤：
1. 查看聚合报告，了解整体指标
2. 分析响应时间趋势
3. 检查错误率变化
4. 监控服务器资源
5. 定位性能瓶颈

### 场景设计

**Q: 如何设计性能测试场景？**

A: 常见场景：
- 基准测试：单用户执行，获取基准数据
- 并发测试：逐步增加并发用户
- 稳定性测试：持续运行8-24小时
- 峰值测试：模拟业务峰值
- 极限测试：逐步增加到系统崩溃

**Q: 什么是梯度加压？**

A: 梯度加压是逐步增加并发用户数，观察系统在不同负载下的表现，找到系统的性能拐点和最大承载能力。

```
示例梯度：
0-5分钟: 10个用户
5-10分钟: 20个用户
10-15分钟: 50个用户
15-20分钟: 100个用户
20-25分钟: 200个用户
```

## 3. JMeter 文档

### JMeter 核心组件

| 组件 | 作用 | 常用类型 |
|------|------|----------|
| 线程组 | 模拟用户 | Thread Group, Stepping Thread Group |
| 取样器 | 发送请求 | HTTP Request, JDBC Request |
| 监听器 | 查看结果 | Summary Report, Aggregate Report |
| 断言 | 验证结果 | Response Assertion, JSON Assertion |
| 配置元件 | 配置参数 | CSV Config, HTTP Header Manager |
| 定时器 | 控制间隔 | Constant Timer, Gaussian Timer |
| 前置处理器 | 请求前处理 | User Parameters |
| 后置处理器 | 响应后处理 | JSON Extractor, Regex Extractor |

### 常用函数

```
${__time(yyyy-MM-dd HH:mm:ss)}  # 当前时间
${__RandomString(10,abcdefg)}    # 随机字符串
${__Random(1,100)}               # 随机数
${__CSVRead(test.csv,0)}         # 读取CSV
${__P(property)}                 # 读取属性
${__threadNum}                   # 线程号
${__iterationNum}                # 迭代次数
```

### 断言示例

```javascript
// JSON断言
{
  "code": 200,
  "message": "success",
  "data": {
    "id": "${__Random(1,1000)}"
  }
}

// 响应断言
- 测试字段: Response Code
- 模式匹配: Equals
- 测试模式: 200

// 大小断言
- 响应大小大于: 100 bytes
```

## 4. 性能测试报告模板

```markdown
# 性能测试报告

## 1. 测试概述
- 测试时间: 2024-01-15 14:00 - 17:00
- 测试环境: 生产环境
- 测试工具: JMeter 5.6.2

## 2. 测试结果

### 2.1 响应时间
| 接口 | 平均(ms) | 90%(ms) | 99%(ms) | 最大(ms) |
|------|----------|---------|---------|----------|
| 登录 | 120 | 180 | 250 | 500 |
| 查询 | 85 | 120 | 180 | 350 |
| 提交 | 200 | 300 | 450 | 800 |

### 2.2 吞吐量
| 并发用户 | TPS | 响应时间(ms) | 错误率 |
|----------|-----|-------------|--------|
| 100 | 500 | 120 | 0% |
| 200 | 800 | 180 | 0% |
| 500 | 1200 | 350 | 0.5% |
| 1000 | 1500 | 600 | 2% |

### 2.3 资源使用
| 指标 | 平均 | 峰值 | 建议 |
|------|------|------|------|
| CPU | 45% | 78% | 正常 |
| 内存 | 62% | 85% | 需关注 |
| 磁盘IO | 30% | 55% | 正常 |

## 3. 结论
- 系统在500并发用户下表现良好
- 超过800并发用户后响应时间明显增加
- 建议优化数据库查询和缓存策略

## 4. 优化建议
1. 优化慢SQL查询
2. 增加Redis缓存
3. 增加应用服务器节点
```
