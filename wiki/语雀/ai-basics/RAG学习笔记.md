## 一、RAG基础知识

### (一) RAG是什么？

RAG（Retrieval-Augmented Generation, RAG）是一种技术框架或设计模式。其核心思想是：在让大语言模型（如GPT、Llama等）生成回答之前，先从一个外部知识库中检索出与用户问题相关的信息，然后将这些信息作为上下文，连同原始问题一起"喂"给大模型，最后由大模型基于这些资料生成最终答案。

简言之，它解决了让大模型在生成答案时能参考最新、最准确、最相关的外部资料的问题。

### (二) 解决的问题

传统的直接使用大模型（称为"闭卷考试"模式）存在显著缺陷：

- **知识截止：** 模型的知识停留在其训练数据的时间点，无法获取最新信息
- **幻觉：** 在面对不确定或未见过的问题时，模型可能"一本正经地胡说八道"（因为大模型的执行过程是在不断的预测下一个最高概率的词（token）。在仅有预训练数据且上下文不足的情况下，大模型就会出现'幻觉'）
- **数据私有性：** 企业内部的敏感数据无法用于模型训练，但业务又需要模型理解这些数据
- **领域专业性：** 通用模型在特定深度垂直领域（如医疗、法律、金融）的知识深度不足

RAG 通过引入外部检索，有效缓解了以上所有问题。

### (三) 工作流程

RAG 的经典工作流程是一个三步曲：索引、检索、生成。

**1. 索引**
- 将知识库文档（PDF、Word、网页等）拆分成小段落或句子
- 使用文本嵌入模型（Embedding Model）将每个文本片段转换成一个高维的数字向量
- 将这些向量及其对应的原始文本存储起来（向量数据库）

**2. 检索**
- 当用户提问时，同样使用文本嵌入模型将问题也转换成一个向量
- 在向量数据库中，快速计算这个问题向量与所有知识库文本向量的相似度（如余弦相似度）
- 返回与问题最相关（即相似度最高）的 Top-K 个文本片段

**3. 生成**
- 将检索到的相关文本片段（Context）和用户的原始问题（Prompt）组合成一个增强的提示词
- 将这个提示词发送给大语言模型（LLM）
- LLM 基于这些确凿的参考资料，生成一个更准确、可溯源、更少幻觉的答案

## 二、检索器 Retriever

Retriever 的任务是：在用户问题进入大模型之前，从外部知识库中找出最相关、最可信、最适合放进上下文的内容。

### (一) Retriever 是什么？

Retriever（检索器）是 RAG 系统中负责从知识库中查找相关内容的模块。

它可以基于多种方式工作：

- **Keyword Search（关键词搜索）** — 典型算法：BM25，擅长精确词、错误码、接口名、法条编号、产品 SKU 等查询
- **Semantic Search（语义搜索）** — 基于 Embedding（向量嵌入），擅长自然语言问题、同义表达、语义相近但字面不一致的查询
- **Hybrid Search（混合搜索）** — 同时使用 BM25 和向量搜索，兼顾关键词精确性和语义召回能力
- **Metadata Filtering（元数据过滤）** — 根据结构化字段过滤文档，例如：部门、权限、时间、文档类型、租户、语言、产品线等
- **Re-ranking（重排序）** — 使用更强但更慢的模型对初步检索结果重新排序，常见模型：Cross-Encoder、BGE Reranker、Cohere Rerank 等

### (二) 关键词搜索：BM25

BM25（Best Matching 25）是经典的信息检索排序算法，广泛用于搜索引擎、文档检索和 RAG 系统中。它属于 Lexical Search（词法搜索）或 Sparse Retrieval（稀疏检索）。

**BM25 的核心思想：** 如果一个文档中包含用户查询中的重要词，并且这些词在整个语料库中较少见，那么这个文档更可能相关。

**BM25 的三个核心因素：**

**1. 词频-TF（Term Frequency）**
某个查询词在文档中出现得越多，通常相关性越高。但 BM25 使用 TF Saturation（词频饱和）：一个词出现 100 次，并不意味着它比出现 1 次重要 100 倍。

**2. 逆文档频率-IDF（Inverse Document Frequency）**
如果一个词在很多文档中都出现，那么它区分文档的能力较弱。BM25 会给稀有词更高权重。

**3. 文档长度归一化（Document Length Normalization）**
长文档天然更容易包含更多关键词。BM25 会对文档长度进行调整，避免很长的文档仅仅因为包含更多词而压过短小但精准的文档。

**BM25 的优势场景：**

| 场景 | 示例 |
|------|------|
| 错误码检索 | ORA-12541、ERR_0427 |
| API 文档检索 | POST /v2/exports |
| 法律条款检索 | section 7.2、GDPR Article 17 |
| 代码检索 | BillingService::retryCharge() |
| 产品编号 | SKU-AB123 |
| 精确短语 | "dead-letter queue threshold" |

**BM25的局限：** 无法理解语义，也无法结合上下文识别同义词，对同义词、改写表达、语义相关但字面不同、跨语言表达、用户不知道准确术语的自然语言查询不够强。

### (三) 语义搜索：Semantic Search

**1. Semantic Search 是什么？**

Semantic Search（语义搜索）是基于语义相似度的检索方式。它通常使用 Embedding Model（嵌入模型）、Vector Database（向量数据库）、ANN（近似最近邻搜索）。

**2. Bi-Encoder：双编码器**

语义检索中常见的是 Bi-Encoder（双编码器）架构。它会分别编码用户查询和文档 chunk，然后通过向量相似度计算相关性。

常见相似度计算方式：
- Cosine Similarity（余弦相似度）
- Dot Product（点积）
- Euclidean Distance（欧氏距离）

**Bi-Encoder 的好处：**
- 文档向量可以提前计算
- 查询时只需要计算 query vector
- 可以高效处理百万、千万级文档

**问题：** 它会把一个 chunk 压缩成一个固定长度向量，压缩过程中会丢失一些精确信息。

**语义搜索的优势场景：**

| 场景 | 示例 |
|------|------|
| 自然语言问题 | "怎么提升系统稳定性？" |
| 同义表达 | "终止合同" vs "取消协议" |
| 概念检索 | "如何处理生产事故？" |
| 用户不知道术语 | "服务挂了之后怎么自动恢复？" |
| 多语言近似匹配 | 中文问题检索英文文档 |

**语义搜索的局限：** 不擅长精确错误码、API 路径、函数名、版本号、法条编号、SKU、特定专有名词。

### (四) 元数据过滤：Metadata Filtering

**1. Metadata 是什么？**

Metadata（元数据）是描述文档或 chunk 的结构化信息。在 RAG 中，元数据非常关键，它不是用来生成答案的主要文本，而是用来控制：搜索范围、权限边界、业务上下文、数据时效性、多租户隔离、合规与审计。

**2. 元数据过滤的作用**

- **降低搜索空间：** 元数据过滤可以在执行向量搜索前缩小检索空间，使候选池更小、更相关
- **防止越权访问：** 语义相似度不是访问控制机制，应该在检索之前进行过滤
- **避免过期文档：** 可以加入时间过滤，避免旧信息浮现
- **多租户隔离：** SaaS 场景中，每个客户的数据必须隔离

**3. Pre-filter 与 Post-filter**

| 类型 | 含义 | 优点 | 缺点 |
|------|------|------|------|
| Pre-filter | 检索前过滤候选集 | 快、安全、减少成本 | 过滤过严可能漏掉答案 |
| Post-filter | 检索后过滤结果 | 不影响初始召回 | 可能浪费计算，且 top-k 被过滤后不足 |

生产系统通常优先使用 Pre-filter（预过滤），尤其在权限、安全、租户隔离场景中。

**4. 元数据过滤的风险：** 如果过滤条件太严格，可能把正确文档排除掉。

### (五) 混合搜索：Hybrid Search

**1. 为什么需要混合搜索？**

| 检索方式 | 优势 | 缺点 |
|---------|------|------|
| BM25 | 精确、可解释、适合编号/错误码/API | 不懂语义、同义词差 |
| 向量搜索 | 懂语义、适合自然语言 | 精确标识符弱、可解释性差 |
| 混合搜索 | 兼顾精确匹配和语义召回 | 架构更复杂、需要调参 |

**2. 混合搜索的结果融合策略**

**Weighted Sum（加权求和）：** 最简单的方式，但要求先做分数归一化

**Alpha 参数：** 很多系统会使用 Alpha（α）表示语义搜索和关键词搜索的权重

| 查询类型 | 推荐倾向 |
|---------|---------|
| 概念性问题 | 提高向量权重 |
| 错误码/API/编号 | 提高 BM25 权重 |
| 法律条款 | 提高 BM25 权重 |
| 客服自然语言 | 提高向量权重 |
| 企业技术文档 | 通常 0.35 ~ 0.65 之间实验 |

**RRF（Reciprocal Rank Fusion，倒数排名融合）：** 不直接融合原始分数，而是融合排名。优势是不需要 BM25 和向量分数同尺度，对不同检索器结果稳定。

### (六) Re-ranking：重排序

**1. 为什么需要重排序？**

Hybrid Search 可以扩大候选集，但初始排序仍可能不够精准。生产 RAG 中常见架构是"先粗召回，再精排序"。

**2. Cross-Encoder：交叉编码器**

Cross-Encoder 会把 query 和 document 拼接在一起输入模型，让它直接判断二者是否相关。它可以更细粒度地捕捉：查询中的具体实体、文档中的具体答案、数字、条件、限定词、query-document token 之间的交互关系。

Cross-Encoder 比 Bi-Encoder 更适合精确排序，但它不能预计算所有文档，因此不能直接用于全库搜索，只适合对 Top-N 候选进行重排序。

**3. Re-ranking 的收益**

重排序通常提升的是：Context Precision（上下文精确率）、Answer Relevancy（答案相关性）、Faithfulness（忠实度 / 事实一致性）

### (七) 搜索质量评估

**1. 为什么要评估 Retriever？**

RAG 的问题经常不是 LLM 不行，而是 Retriever 没把正确上下文传给 LLM。Retrieval 决定了 RAG 答案是否可信。

**2. Retriever 常用评估指标**

- **Precision@K（精确率）：** 评估返回给 LLM 的上下文是否干净，有多少比例的文件是有用且可靠的
- **Recall@K（召回率）：** 评估正确答案有没有被检索出来，检索出来的文件包含多少百分比的正确答案文件
- **Hit Rate@K（命中率）：** 只要 Top-K 中包含至少一个正确文档，就算命中
- **MRR（Mean Reciprocal Rank，平均倒数排名）：** 关注第一个正确结果出现的位置，适合评估正确答案是否排得足够靠前

### (八) Chunking 对 Retriever 的影响

Retriever 的效果不只取决于搜索算法，也强烈依赖 chunk 的质量。

**Chunk 太大的问题：**
- 向量表示被稀释
- 一个 chunk 包含多个主题
- 精确答案被大量无关内容淹没

**Chunk 太小的问题：**
- 上下文不完整
- BM25 短语匹配被切断
- LLM 拿不到足够背景

**结构化切分更好：** 按标题、段落、列表、代码块、表格、法条、FAQ 问答对等结构边界切分。

### (九) 总结

高质量 Retriever 至少要考虑：

1. **BM25 关键词检索** — 负责精确匹配，适合错误码、API、编号、法律条款等
2. **语义向量检索** — 负责理解自然语言意图，适合同义词、改写表达和概念查询
3. **元数据过滤** — 负责权限、租户、时间、部门、产品等结构化约束，是企业 RAG 安全和准确性的基础
4. **混合搜索** — 将关键词精度和语义召回结合起来，常用 Weighted Sum、Alpha 调权、RRF 等方法
5. **重排序** — 用 Cross-Encoder 等模型精排 Top-N 候选，显著提升上下文质量
6. **搜索质量评估** — 用 Precision@K、Recall@K、MRR、NDCG@K 等指标持续评估

**LLM 决定了答案表达能力，Retriever 决定了答案事实基础。RAG 做得好不好，首先看检索器能不能把正确上下文稳定、快速、可解释地送到模型面前。**
