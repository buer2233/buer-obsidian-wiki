## 一、使用前准备

### Claude Code安全提醒

**Claude官方提醒：**
1. Claude可能会犯错误，您应该经常检查Claude的响应，特别是在运行代码时。
2. 由于提示注入的风险，只对你信任的代码使用它。更多详细信息请参见：https://code.claude.com/docs/en/security

**开源包被投毒的风险案例：**
1. LiteLLM投毒（公司遇到的安全事故）
2. axios投毒（使用龙虾安装环境中招，导致需要整机重装系统）

这两个案例都是黑客向使用量很高的最新版本的开源包投毒。而使用AI帮我们安装环境时，AI都会默认安装最新版本的依赖包。这就导致使用AI遇到这种风险的概率更高，这也提醒我们使用AI需要谨慎。

## 二、工具和账号配置

### 1. 安装 Claude Code（Windows）

#### 1.1 安装前检查 Node.js，版本需要在22以上

1. 打开 PowerShell
2. 执行 `node -v`
3. 如果能看到版本号，例如 v20.x.x，说明 Node.js 已安装
4. 如果提示未识别 node 命令，请先安装 Node.js：
   - 下载地址：https://nodejs.org/
   - 建议安装 LTS 版本
   - 安装完成后重新打开 PowerShell，再执行 node -v 验证

#### 1.2 安装 Claude Code

1. 打开 PowerShell
2. 执行安装命令：`npm install -g @anthropic-ai/claude-code`
3. 安装完成后执行：`claude --version`
4. 如果能看到 Claude Code 版本号，说明安装成功
5. 如果报错找不到命令，大概率是未添加环境变量

#### 1.3 Claude Code的首次配置

1. 修改 `~/.claude/settings.json`（没有文件可手动新建）
2. 修改完成后在任何目录使用powershell执行命令：`claude`，能正常进入界面就是成功了

### 2. 安装 CC Switch

CC Switch 用于管理 Claude Code 的不同模型服务配置，本文使用它配置 DeepSeek。

#### 2.1 下载 Windows 安装包

1. 打开浏览器访问 CC Switch 发布页
2. 页面打开后，拉到最下面的 Assets 区域
3. 点击 CC-Switch-Windows.msi 下载

#### 2.2 安装 CC Switch

1. 双击下载的 CC-Switch-Windows.msi
2. 按安装向导点击下一步
3. 安装路径如无特殊要求，保持默认即可
4. 安装完成后，在开始菜单或桌面快捷方式中打开 CC Switch

### 3. 在 PyCharm 中安装 CC GUI 插件

CC GUI 插件用于在 PyCharm 中调用 Claude Code / CC Switch 配置，方便直接在项目内编写接口自动化用例。

#### 3.1 打开 PyCharm 插件市场

1. 打开 PyCharm
2. 打开接口自动化项目
3. 进入菜单：File → Settings（或使用快捷键 Ctrl + Alt + S）
4. 在左侧选择：Plugins
5. 切换到：Marketplace

#### 3.2 搜索并安装 CC GUI

1. 在插件搜索框输入：CC GUI
2. 找到对应插件后点击 Install
3. 安装完成后，按 PyCharm 提示重启 IDE
4. 重启后检查 PyCharm 侧边栏、底部工具窗口或菜单中是否出现 CC GUI 入口
5. 检查node.js是否配置，并且版本大于等于22
6. 检查SDK是否安装，且版本建议更新到最新版本

### 4. 注册并配置 DeepSeek 开发者账号

#### 4.1 注册 DeepSeek 开发者账号

1. 打开浏览器访问 DeepSeek 平台
2. 点击注册或登录
3. 按页面提示完成手机号、邮箱或第三方账号登录
4. 登录后进入 DeepSeek 开发者控制台

#### 4.2 创建 API Key

1. 在 DeepSeek 开发者控制台中找到 API Key 管理入口
2. 创建新的 API Key
3. 创建后立即复制并保存 API Key
4. **请勿把 API Key 提交到 Git、发到群聊或写进测试代码**

### 5. 将 DeepSeek 配置到 CC Switch

#### 5.1 DeepSeek 配置

配置信息参考：https://api-docs.deepseek.com/zh-cn/quick_start/agent_integrations/claude_code

#### 5.2 验证 Claude Code 使用 DeepSeek

1. 打开 PowerShell
2. 进入自己的接口自动化项目目录
3. 启动 Claude Code
4. 输入简单问题验证
5. 如果能正常回复，说明 Claude Code + CC Switch + DeepSeek 链路可用
6. 执行初始化命令生成CLAUDE.md文件

#### 5.3 常见问题

**问题：401 / Unauthorized**
处理：检查 API Key 是否复制完整，是否有多余空格，DeepSeek 账号是否可用。

**问题：模型不存在**
处理：检查模型名是否填写为 deepseek-chat 或团队指定模型名。

## 三、接口自动化用例编写

### 1. 准备 api-test-E10 Skill

#### 1.1 手动复制Skill 文件到claude目录下

#### 1.2 确认接口自动化项目结构

1. 用 PyCharm 打开接口自动化项目
2. 确认项目中存在接口自动化目录
3. 确认 config.py 中当前启用的 RunConfig.baseurl 与浏览器访问的被测环境域名一致

#### 1.3 在 Claude Code / CC GUI 中触发 Skill

在 CC GUI 对话中说明要使用 api-test-E10 Skill

### 2. 使用 CC GUI 编写接口自动化用例

#### 2.1 打开 CC GUI

1. 打开 PyCharm
2. 打开接口自动化项目
3. 打开 CC GUI 工具窗口
4. CC GUI设置-供应商管理-选择使用本地配置信息
5. CC GUI设置-SKD依赖，确认Claude Code SDK可用

#### 2.2 按任务类型提供前置信息

正式新增或维护接口方法、接口用例前，先确认本次任务类型。新增任务必须提供 5 项；维护任务只强制提供 2 项。

#### 2.3 重要填写规则

- [接口方法文件] 与 [接口方法位置] 必须同时填写真实内容，或同时填写"当前用例无新增接口"
- 不能保留"填写接口方法所在文件路径"这类占位符原文
- [fixture] 为选填项，可省略或留空，不参与缺项判定
- 新增任务的 [用例名] 要写完整中文功能名，不要只写英文缩写或简单编号
- 维护任务不强制要求 [接口方法文件] / [接口方法位置] / [用例名]，除非定位后确认必须修改接口方法且无法从现有用例反查
- 如果是纯查询、检查环境、启动抓包，不需要填写任务信息
- 一旦进入"新增 / 维护接口方法或用例"阶段，必须按任务类型补齐对应信息

### 3. 用例编写/维护方式总览

api-test-E10 Skill 支持新增四种方式、维护四种方式。

#### 新增任务四种方式

| 方式 | 名称 | 适合场景 | 需要你提供 |
|------|------|---------|-----------|
| 方式 1 | 抓包驱动 | 新接口多、业务链路复杂、希望从真实 UI 操作分析并设计用例 | 抓包环境、UI 操作、勾选接口 |
| 方式 2 | 参考已有用例 | 已有相似用例，只需仿写、改参数、改断言 | 参考用例路径或函数名、差异点 |
| 方式 3 | cURL 手工 | 抓包不可用、接口数量少、能手动复制请求响应 | cURL、响应体、业务说明 |
| 方式 4 | Java Controller 源码参考 | 后端已有接口定义，希望按 Controller/Jacoco 找出接口自动化未覆盖接口并补齐用例 | Controller 源码、Jacoco 报告链接或本地 Controller Markdown 文件 |

#### 维护任务四种方式

| 方式 | 名称 | 适合场景 | 需要你提供 |
|------|------|---------|-----------|
| 方式 1 | 抓包驱动 | 业务链路变化较大、多接口联动、需要同步最新请求路径 | 最新抓包或 UI 操作 |
| 方式 2 | 参考已有用例 | 同类用例结构稳定，只改参数、fixture、断言或调用方式 | 参考用例或同类样本 |
| 方式 3 | cURL 手工 | 少量接口变更明确，不需要完整抓包回溯 | cURL、响应体、差异说明 |
| 方式 4 | pytest 报错驱动 | 只指定待维护用例，让 AI 直接跑 pytest，按最后一个中断报错分类 | [接口用例文件] 和 [接口用例位置] |

### 4. 方式 1：抓包驱动编写用例

#### 4.1 方式 1 适用场景

- 需要覆盖一条完整业务链路
- 页面操作会触发多个接口
- 不确定哪些接口需要新增方法
- 希望 AI 从真实请求中识别新接口、已实现接口和特殊接口

#### 4.2 首次使用前配置抓包环境

按 capture/README.md 完成以下事项：
1. 确认 Python 版本
2. 安装 mitmproxy
3. 验证 mitmproxy
4. 启动抓包
5. 配置浏览器代理：地址 127.0.0.1，端口 12138
6. 访问 http://mitm.it 下载并安装 Windows 证书
7. 证书必须安装到：本地计算机 - 受信任的根证书颁发机构
8. 打开被测系统完成一次操作，确认生成抓包数据

#### 4.3 方式 1 提示词填写步骤

**步骤 1：先发送任务信息**
在 CC GUI 中发送任务信息

**步骤 2：让 AI 检查或启动抓包**
继续发送抓包相关指令

**步骤 3：你在浏览器中完成业务操作**
1. 确认浏览器代理已切到 127.0.0.1:12138
2. 打开被测系统
3. 按用例需要完成一遍 UI 操作
4. 操作完成后回到 CC GUI
5. 发送操作完成通知

**步骤 4：勾选需要生成用例的接口**
AI 会生成勾选草稿，你需要：
1. 打开勾选草稿
2. 保留需要写入用例的接口为 [x]
3. 不需要的接口改成 [ ]
4. 保存文件
5. 回到 CC GUI 发送确认

**步骤 5：根据 AI 反馈补充信息**
如果 AI 提示以下内容，请按提示补充：
- 某个接口是否需要复用已有方法
- 某个字段断言规则不明确
- 某个接口响应体过大或为二进制
- 登录态、租户、组织、用户、流程数据需要你确认

### 5. 方式 2：参考已有用例编写

#### 5.1 方式 2 适用场景

- 已经有相似业务用例
- 新用例只是改页面、改参数、改断言
- 不需要新增接口方法
- 同一类用例需要批量铺开

#### 5.2 方式 2 提示词填写步骤

**步骤 1：找到参考用例**
在 PyCharm 中找到你想参考的已有用例，记录：
- 参考用例文件路径
- 参考用例函数名
- 新旧用例差异点

**步骤 2：判断是否新增接口方法**
如果完全复用已有接口方法，填写"当前用例无新增接口"
如果需要新增接口方法，则填写真实文件和位置

**步骤 3：发送方式 2 提示词**

**步骤 4：按 AI 提问补充业务差异**
AI 可能会要求补充：
- 新用例使用的数据来源
- 是否需要前置创建数据
- 是否需要清理数据
- 断言字段和预期值
- 是否复用参考用例中的 fixture

### 6. 方式 3：cURL 手工编写

#### 6.1 方式 3 适用场景

- 抓包服务暂时不可用
- 抓包数据太多，不方便筛选
- 接口数量较少，可以手动复制 cURL

7. 右键请求

9. 打开请求的 Response 或 Preview，复制接口响应体

**步骤 1：整理接口信息**
每个接口建议按以下格式整理，如果有多个接口，按接口 1、接口 2、接口 3 依次排列

**步骤 2：发送方式 3 提示词**

**步骤 3：补充字段说明**
如果 cURL 或响应中有业务字段不容易判断，请额外补充说明

### 7. 方式 4：Java Controller 源码参考编写

#### 7.1 方式 4 适用场景

- 后端已经有 Java Controller 源码、Controller Markdown 文件或 Jacoco Controller HTML 报告
- 希望从后端接口定义中找出"接口自动化未覆盖"的接口
- 不方便抓包，或者抓包只能覆盖部分页面操作
- 想按 Controller 的业务接口分组补齐 _CSC.py 专用用例

#### 7.2 方式 4 需要准备的信息

方式 4 仍然属于"新增任务"，所以必须先提供新增任务 5 项信息

同时准备以下任意一种 Controller 输入：
- Jacoco Controller HTML 报告链接，推荐优先使用
- 本地 Java / Markdown 文件
- 直接粘贴 Java Controller 源码

说明：Jacoco 覆盖颜色只能作为参考。接口是否已经被接口自动化覆盖，最终以项目中的 tools/page_api_index.sqlite3 按 api_url + method 查重结果为准。

#### 7.3 _CSC.py 用例文件建议

方式 4 推荐把新增用例写到 _CSC.py 结尾的专用文件中，CSC 表示 controller_sourceCode。

#### 7.4 方式 4 提示词填写步骤

**步骤 1：发送任务信息和 Controller 来源**

**步骤 2：等待 AI 生成分析草稿**
AI 会执行分析命令，分析结果默认生成到项目根目录

分析草稿通常会包含：
- 从类级 @RequestMapping 和方法级 @GetMapping / @PostMapping 等注解拼出的完整接口 URL
- 按 api_url + method 查到的已覆盖接口和未覆盖接口
- 覆盖状态异常提醒
- 参数来源特殊、可能需要前置数据、暂不适合自动化的接口提醒
- 建议测试场景和业务分组

**步骤 3：你调整分析草稿**
按实际需要调整：
- 保留需要生成用例的接口为 [x]
- 暂不处理的接口改成 [ ]
- 可以移动接口到其他场景
- 可以修改场景标题
- 可以补充参考用例备注

**重要规则：**
- 未覆盖接口默认 [x] 只是候选池，不代表要全部塞进一条用例
- 必须等你确认草稿后，AI 才能继续生成接口方法和 pytest 用例

**步骤 4：确认草稿后让 AI 继续**

AI 后续会按以下顺序执行：
1. 读取你调整后的分析草稿
2. 只处理保留 [x] 的接口和场景
3. 根据接口名、@ApiOperation、参数类型、返回语义、service 调用等信息分析调用链路
4. 主动检索相似用例，复用已有 fixture、前置数据、payload 拼接和断言风格
5. 已覆盖接口复用现有接口方法，未覆盖接口按你填写的位置新增
6. 按场景分组写 _CSC.py 用例，不机械按接口列表顺序堆接口
7. 第一阶段只做基础成功断言并打印完整返回
8. 跑 pytest 获取真实返回
9. 第二阶段基于真实返回补充结构和关键字段断言
10. 再次跑 pytest 闭环验证

#### 7.6 方式 4 常见注意事项

- Java Controller 只能说明接口入口和参数大致来源，不能保证 payload 和真实响应完全正确
- payload 优先参考相似用例、已有接口方法、真实项目调用习惯
- 断言以 pytest 真实返回为准，不要只按 Java 返回类型硬写
- 如果基于源码信息调试 3 次仍无法通过，AI 应停止继续尝试
- 如果 Jacoco 显示已覆盖但接口索引未覆盖，或 Jacoco 显示未覆盖但接口索引已覆盖，需要在结果中提醒你排查覆盖数据来源

### 8. 编写完成后的验证步骤

#### 8.1 最小范围运行 pytest

1. 在 PyCharm 中查看 Git 变更
2. 确认只修改了本次相关文件
3. 确认没有提交运行期产物，例如：
   - capture/latest.jsonl

### 9. 推荐使用流程

**首次配置时按以下顺序执行：**
1. 安装 Node.js
2. 安装 Claude Code
3. 安装 CC Switch
4. 在 DeepSeek 平台注册账号并创建 API Key
5. 在 CC Switch 中新增 DeepSeek 配置
6. 在 PyCharm 中安装 CC GUI 插件
7. 打开接口自动化项目
8. 在 CC GUI 中验证 AI 可正常回复
9. 确认 api-test-E10 Skill 可触发
10. 首次使用方式 1 时，按 capture/README.md 配置 mitmproxy、证书和浏览器代理
11. 按本教程"三、接口自动化用例编写"的第 4、5、6、7 节选择一种方式编写用例
12. 执行最小范围 pytest 验证

**日常编写用例时按以下顺序执行：**
1. 打开 PyCharm 和 CC GUI
2. 先确认本次是新增任务还是维护任务
3. 新增任务准备 5 项任务信息；维护任务准备 2 项维护任务信息
4. 新增任务选择方式 1、方式 2、方式 3 或方式 4；维护任务选择方式 1、方式 2、方式 3 或方式 4
5. 按对应模板发送提示词
6. 根据 AI 提问补充业务细节
7. 查看生成代码
8. 运行或确认 AI 已运行最小范围 pytest
9. 处理失败日志直到通过
10. 检查 Git 变更，避免提交运行期产物

### 10. 常见问题

#### 10.1 Claude Code 能打开，但 CC GUI 不可用

处理步骤：
1. 确认 PyCharm 已安装 CC GUI 插件
2. 重启 PyCharm
3. 检查插件入口：View → Tool Windows
4. 确认 CC Switch 已切换到 DeepSeek 配置

#### 10.2 AI 没有按 api-test-E10 规范执行

处理：在提示词开头明确写"请使用 api-test-E10 Skill"

#### 10.3 提示缺少任务信息

处理：先确认任务类型，再检查对应信息。

**新增任务检查是否完整填写了 5 项：**
- [接口方法文件]
- [接口方法位置]
- [接口用例文件]
- [接口用例位置]
- [fixture]（选填，可省略或留空）
- [用例名]

**维护任务检查是否完整填写了 2 项：**
- [接口用例文件]
- [接口用例位置]

#### 10.4 抓包服务启动失败

处理：按 capture/README.md 排查，重点检查：
- Python 是否安装
- mitmdump --version 是否可用
- 12138 端口是否被占用
- 证书是否安装到本地计算机的受信任的根证书颁发机构
- 浏览器代理是否为 127.0.0.1:12138

#### 10.5 不知道选哪种编写方式

建议：
- 有真实页面操作、新链路复杂：优先选方式 1
- 有非常相似的已有用例：优先选方式 2
- 只有接口请求和响应，没有抓包环境：选方式 3
- 有 Controller 源码、Jacoco 报告，想从后端接口定义补齐未覆盖接口：选新增方式 4
- 维护已有用例且希望 AI 自行运行 pytest 看报错修复：选维护方式 4
